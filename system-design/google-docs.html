01Problem & Requirements

The ask. Real-time collaborative document editor — multiple users edit the same doc, see each other's changes within a second, never lose a keystroke, work offline.
Functional
  • Multiple users edit the same document concurrently (text + basic formatting).
  • Changes propagate to all other viewers within ~1 second.
  • Persistent — close the tab, come back, doc is still there with full history.
  • Presence — see who else is in the doc (avatars, live cursors).
  • Offline edits sync once the user reconnects.
  • Permissions — owner, editor, commenter, viewer.
Non-Functional
  • ~100M DAU, average user has 10 docs, 1% are co-edited simultaneously.
  • Latency: local echo < 50 ms, peer visibility p99 < 1 s.
  • Availability > Consistency — never block typing; converge eventually.
  • Durability: never lose a committed keystroke (11 nines).

02Capacity Math

Notation. b = bit, B = byte, 1 B = 8 b. 1 KB = 10³ B, 1 MB = 10⁶ B, 1 GB = 10⁹ B, 1 TB = 10¹² B. 1 char = 1 B (UTF-8 ASCII).
  1. Active concurrent editors = 100M DAU × 1% co-edit = 10⁶ concurrent sessions.
  2. Edit ops/sec = 10⁶ × 4 chars/sec × 20% active = ~10⁶ ops/sec (≈1M ops/sec).
  3. Per-op payload (CRDT op: position + char + clientId + ts) ≈ 60 B.
  4. Edit ingress bandwidth = 10⁶ × 60 B = 60 MB/s.
  5. Fan-out egress (each op → ~3 peers) = 60 MB/s × 3 = 180 MB/s.
  6. Storage: 10⁹ docs × 50 KB avg = 50 TB body + 100 TB op log + 50 TB snapshots = ~200 TB; replicated RF=3 → ~600 TB raw.

03Why "Concurrent Editing" is the Hard Part

If Alice and Bob both type at the same position at the same time, naïve last-write-wins loses one of them. The system must let every replica apply ops in any order and converge to the same final document.

ApproachHow it worksTradeoff
Operational Transformation (OT) Ops are insert(pos, char) / delete(pos, len). Server transforms each op against intervening ops to keep positions valid. Requires a central server to serialize transforms. Original Google Docs design.
CRDT (sequence CRDT, e.g. RGA / Yjs / Automerge) Each character gets a globally unique, totally ordered ID ((clientId, lamport, fractional-pos)). Ops are commutative — any order works. No central serialization needed. Composes with offline edits. Modern Google Docs, Figma, Notion.
Pick CRDT. The server stops being the source of truth for ordering — it becomes a relay + persistence layer. Offline edits "just work". Cost: per-character metadata + tombstones for deleted chars (CRDTs grow over time → snapshot compaction matters).
CRDT (Conflict-free Replicated Data Type) = a data structure where concurrent updates from different replicas always converge to the same state without coordination. For text editing, a sequence CRDT assigns each character a unique ID so inserts at the same position have a deterministic tie-break.
Tombstone = a "deleted" marker the CRDT keeps so a concurrent insert after a deleted character still anchors correctly. Tombstones grow forever unless snapshot compaction removes them.

Collab Server = a stateful pod that owns the live CRDT replicas for a set of documents. Every editor of a single doc routes to the same Collab Server pod — fan-out happens inside one process. Pod crash = ~5 s reconnect + replay from snapshot.
Redis Pub/Sub for cross-pod relay = when two editors of the same doc somehow land on different pods (region failover, brief sticky-session miss), the secondary pod subscribes to doc:{doc_id} on Redis to receive ops from the home pod. The home pod is still the source of truth; Redis Pub/Sub is just a fast best-effort bridge.

05Happy-Path Flow — One Keystroke

  1. Alice types "X". Her browser CRDT generates {type:"ins", id:(clientA, lamport=42), after:"id-9", char:"X"}.
  2. Local echo: CRDT applies the op locally first → "X" renders within 50 ms. Never blocks on network.
  3. Op sent over the persistent WebSocket to the Collab Server pod that owns this doc.
  4. Collab Server:
    • Applies op to in-memory CRDT replica.
    • Publishes to Kafka doc.ops.{shard} for durability.
    • Publishes to Redis doc:{doc_id} for cross-pod fan-out.
    • Broadcasts to every other WebSocket connected to this doc on the same pod.
  5. Bob's browser receives the op via WebSocket, applies to its CRDT. "X" appears.
  6. Async: a persistence worker consumes Kafka → writes op to Cassandra.
  7. Hourly: a snapshotter merges recent ops into the compacted CRDT body in S3, advances snapshot_seq in Spanner, old ops can be truncated.
Alice → Bob end-to-end: typically 100–400 ms, p99 < 1 s. Local echo is independent of network.

Loading a doc
  1. Look up documents.snapshot_seq (latest compacted version).
  2. Fetch the snapshot from S3 (one big read, ~50 KB gzipped).
  3. Replay any doc_ops with op_seq > snapshot_seq (usually < 100 ops).
  4. Hand resulting CRDT to client. Total: ~100 ms.

07Sharding & Routing

Shard key = doc_id (consistent hashing). Every doc maps to exactly one "home" Collab Server pod via a shard map kept in etcd / ZooKeeper.

  • When a user opens a doc, the API Gateway looks up the home pod and sticky-routes the WebSocket there.
  • All editors of one doc share one in-memory CRDT replica → cheap intra-process fan-out.
  • A doc with >500 concurrent editors triggers promotion to a "leader + N relays" topology: the leader serializes ops, relays just fan out to read-mostly viewers.
Why this matters. 99% of fan-out stays inside a single process. Only cross-region scenarios touch Redis Pub/Sub. Network egress stays bounded.
Hot doc risk. One viral file (e.g., the IPO prospectus, a leaked exam) with 10k editors will saturate a single pod's CPU. Mitigation: auto-promote to leader/relay; shed presence updates first (drop cursor frequency from 200 ms → 1 s); never shed actual edit ops.

08Offline Edits & Reconnection

  • Browser keeps an unacked op queue in IndexedDB.
  • When offline, ops still apply locally (CRDT property) — user keeps editing without any spinner.
  • On reconnect:
    1. Client sends last-seen server op_seq and its unacked ops.
    2. Server sends back ops > last_seen_seq.
    3. Client merges — CRDT handles conflicts automatically.
  • Result: 6 hours of in-flight edits on a plane → land → reconnect → everything syncs cleanly, no "merge conflict" dialog.
WAL (Write-Ahead Log) = a local append-only log written before acknowledging an op. The Collab Server keeps a WAL on local disk; on crash, replay the WAL into Kafka so no acked op is lost.

09Permissions, Comments, Cursors

  • ACL check happens at WebSocket-handshake time (Gateway → Auth Service). Cached for the session; revocation broadcasts a kick message over the same channel.
  • Comments are a separate CRDT layer keyed by an anchor range in the doc — when text moves, the anchor moves with it. Stored in doc_comments.
  • Live cursors / presence live entirely in Redis with TTL 30 s — never persisted. Each client broadcasts its cursor position every 200 ms over the same WebSocket.
Presence is ephemeral on purpose. Treating "who's looking at the doc" as durable state means restoring it after every restart, replicating it geographically, and paying storage for data that's stale in 30 seconds. Redis with TTL is the right answer.

12Interview Talking Points

  1. Lead with CRDT vs OT — CRDT removes the server from correctness, enables offline.
  2. Local echo first — typing must never block on the network.
  3. Sticky-route by doc_id — fan-out stays in-process for the common case.
  4. Op log + snapshots — append-only Cassandra log; hourly compaction into S3.
  5. WebSocket for live edits; HTTP only for control plane (open, share, list).
  6. AP over CP — never block typing; converge afterward.
  7. Presence is ephemeral — Redis with TTL, never the database.
  8. Hot docs get promoted to leader/relay topology for 10k-editor viral scenarios.
  9. Trade-offs: CRDT tombstone overhead, snapshot frequency vs replay cost, sticky-session vs even load.
  10. Extensions: suggestion mode, version history UI, real-time chat sidebar, ML grammar service consuming the op stream.

1330-Second Pitch

Google Docs at 1M concurrent editors is fundamentally a CRDT replicated over WebSockets. Each keystroke becomes a CRDT op with a globally unique ID; the client applies it locally first (instant feedback), then ships it over a sticky WebSocket to a Collab Server pod that owns the doc (sharded by doc_id). That pod fans out to every other editor and appends the op to Kafka → Cassandra for durability. Every hour a snapshotter compacts ops into a gzipped CRDT state in S3, so opening a doc = "fetch snapshot + replay tail ops". Offline edits accumulate in IndexedDB and merge cleanly on reconnect — that's the CRDT superpower. AP over CP: never block typing; converge eventually. Presence lives in Redis with a 30 s TTL and never touches the database.

14Glossary

CRDT

Conflict-free Replicated Data Type — concurrent ops from different replicas always converge without coordination.

OT (Operational Transformation)

Alternative to CRDT: a central server rewrites concurrent ops to keep positions consistent.

Lamport timestamp

(counter, clientId) tuple giving a total order across distributed events without clock sync.

Tombstone

A "deleted" marker kept in a CRDT so concurrent inserts after a delete still order correctly.

Sticky session

Load balancer pins one client connection to one backend pod for the lifetime of the session.

WebSocket

Full-duplex TCP-based protocol for low-latency bidirectional messages — used here instead of HTTP polling.

WAL (Write-Ahead Log)

Local append-only log written before acknowledging an op, so a crash can replay anything not yet shipped to Kafka.

Snapshot

Periodically materialized full document state, used to avoid replaying years of ops on doc-open.

Presence

Ephemeral metadata about who's currently viewing or editing (cursors, avatars, colors).

Fan-out

Distributing one incoming event to many connected listeners.

Sharding by doc_id

Every document deterministically maps to one home pod; same idea as Kafka partitioning or DB sharding.

AP vs CP (CAP)

AP = always available, eventually consistent. Docs picks AP because blocking typing is worse than briefly diverging.
