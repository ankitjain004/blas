02Load Balancers & Service Discovery

A load balancer distributes incoming requests across many backend instances. Without it, you can't add capacity by adding machines.

LayerWhat it seesExamplesUse for
L4 (transport)TCP/UDP packets — only IPs and portsAWS NLB, HAProxy in TCP mode, IPVSRaw TCP, gRPC streams, low-latency, anything not HTTP
L7 (application)HTTP headers, URLs, cookies, JWT claimsAWS ALB, Nginx, Envoy, HAProxy in HTTP modePath-based routing, sticky sessions, TLS termination, request transforms
Service mesh sidecarL7 but per-podEnvoy (Istio), LinkerdEast-west traffic inside a Kubernetes cluster
Global / DNS-basedJust hostnamesRoute 53 latency-based routing, Cloudflare LBMulti-region failover, geo-routing
Common balancing algorithms
  • Round-robin — simplest; ignores actual load.
  • Least connections — sends to the backend with fewest open connections; great for variable-duration requests.
  • Weighted — bigger boxes get more traffic. Used during canary rollouts (1% → 10% → 100%).
  • Consistent hashing — same key always lands on the same backend; key for caches and stateful services (see §5 sharding).
  • Power of two choices — pick two random backends, send to the less-loaded one. Provably near-optimal, very cheap.
Sticky session (session affinity) = the LB pins one client to one backend for the lifetime of a session. Used when the backend holds in-memory state for that user. Cost: harder to rebalance load; lose the pod, lose the state.
Service discovery = the mechanism services use to find each other's IPs. Examples: Kubernetes Services + DNS, Consul, etcd, Eureka. The LB and discovery are usually combined in modern stacks.

03API Styles & Gateways

StyleSweet spotWatch out for
REST / JSONPublic APIs, resource CRUD, easy debuggingChatty for nested data; weak typing
gRPC (HTTP/2 + protobuf)Internal service-to-service, polyglot teams, low CPU/network costBrowser support, harder debugging without tools
GraphQLMobile clients that want to pick fields; aggregating many backendsCaching, N+1 queries, complex authz
WebSocket / SSELive bidirectional (chat, docs, dashboards)Sticky LB, harder failover, harder rate-limiting
WebhooksAsync callbacks from your service to a customer's URLRetries, idempotency, signature verification
API Gateway = the single managed entry point in front of many backend services. Owns: TLS termination, authentication, rate limiting, request validation, routing, response transformation, observability. Examples: AWS API Gateway, Kong, Apigee, Envoy at the edge, Cloudflare Workers. In a small system the LB does this; in a microservices system you need a dedicated gateway tier.
BFF (Backend-for-Frontend) = a small gateway-style service per client type (one for web, one for mobile) that aggregates downstream microservices into the exact shape the UI needs. Keeps clients dumb; keeps your microservices uncontaminated by UI concerns.

04Caching Layers

Caches are the cheapest performance lever there is — and the most dangerous source of subtle bugs (stale data, thundering herds, cache stampedes). Senior expectation: pick the right cache layer, name its invalidation strategy.

LayerWhereTTL typicalExamples
Browser / clientEnd-user deviceseconds–hoursCache-Control headers, IndexedDB
CDN / edgeCloudflare/Fastly/Akamai PoPs10s–1hHTML, JSON, images, video segments
API gateway / reverse proxyIn front of your servicessecondsNginx, Varnish, Envoy
Application / distributedBeside your servicesseconds–daysRedis, Memcached, Hazelcast
Local / in-processInside the JVM/Node/Go processsecondsCaffeine (Java), sync.Map+TTL (Go), LRU dict
Materialized viewInside the databaserefresh on schedulePostgres materialized views, BigQuery
Caching strategies
PatternRead pathWrite pathWhen
Cache-aside (lazy)App checks cache → miss → load from DB → populate cacheApp writes to DB; invalidates / updates cacheMost common; works for any DB
Read-throughApp asks cache; cache loads from DB on missSame as cache-asideWhen cache library supports it (e.g., Caffeine)
Write-throughSame as cache-asideApp writes to cache; cache writes through to DB synchronouslyStrong consistency between cache & DB; slower writes
Write-behind (write-back)SameApp writes to cache; cache writes to DB asynchronouslyBurst writes; data loss risk on cache crash
Refresh-aheadBackground refresh near expiry—Hot keys you can't afford to be cold
The four cache failure modes you must mention
  • Cache stampede / thundering herd — TTL expires on a hot key; all requests miss simultaneously and hammer the DB. Fix: single-flight (request coalescing / locking), jittered TTLs, probabilistic early expiration.
  • Stale read — DB updated, cache not invalidated. Fix: write-through, or invalidate-before-update, or accept eventual consistency.
  • Hot key — one key gets 10× the traffic of others, saturates one shard. Fix: replicate hot key across shards, or add a local in-process tier.
  • Negative caching — cache also stores "not found" results to prevent DB scans on bad lookups.
Memcached vs Redis = Memcached is a pure LRU key-value cache (multi-threaded, very simple). Redis is single-threaded but offers data structures (lists, sets, sorted sets, hashes, streams), persistence, scripts, pub/sub, and cluster mode. Default to Redis unless you specifically need Memcached's multi-threaded simplicity.

Wide-columnCassandra, ScyllaDB, HBase, BigtableHuge write throughput, time-series, log-style dataJoins, ad-hoc queries, transactions across rows

GraphNeo4j, JanusGraph, AWS NeptuneRelationship-traversal queries (social graph, fraud rings)Tabular workloads, very wide writes

Time-seriesPrometheus (metrics), InfluxDB, TimescaleDB, ClickHouseRange scans over time, compression, downsamplingRandom by-id lookups

Default for a new service. Postgres until proven otherwise. Add Redis for caching, add a search index when needed, move heavy analytics to ClickHouse/BigQuery, only reach for a distributed DB when single-leader writes truly cap you out. Most "we need NoSQL" instincts are wrong.
OLTP vs OLAP = Online Transaction Processing = lots of small reads/writes by ID (your live app DB). Online Analytical Processing = few big aggregations over huge datasets (your data warehouse). You usually want one of each, connected by a pipeline (CDC, ETL).
CDC (Change Data Capture) = streaming every insert/update/delete out of an OLTP DB as events. Tools: Debezium (Postgres/MySQL binlog), AWS DMS, Materialize. Feeds search indexes, OLAP warehouses, and caches.

06Sharding & Partitioning

One logical dataset split into many physical pieces by some key, so each piece can be served by a different machine. Same pattern applies to databases, caches, queues, and in-memory app servers.

StrategyHowProsCons
Rangee.g., user_id 1–1M → shard 0; 1M–2M → shard 1Range scans are localNew rows pile onto the highest shard ("hotspot at the end")
Hashshard = hash(key) % NEven distributionRange scans go to all shards; resharding moves ~all data
Consistent hashKeys + nodes both placed on a ring; key goes to the next node clockwiseAdding a node moves only ~1/N of keysSlightly less even; needs virtual nodes
Directory / lookupExternal shard map (e.g., Vitess, custom)Full flexibility, can split hot shardsShard map is critical-path metadata; needs HA itself
Geo / localityBy region of the userLatency, data residency (GDPR)Cross-region queries are slow
The shard key is the most important decision
  • Must distribute evenly (no celebrity / Beyoncé problem).
  • Must match the dominant query pattern — if 99% of queries fetch by user_id, shard by user_id.
  • Avoid sharding by anything monotonic (time, auto-incrementing id) without bucketing — you'll create a write hotspot.
  • Resharding is expensive and risky. Pick something stable.
Hot shard = one shard receiving disproportionate traffic. Fixes: replicate that shard more, split it further (sub-shard by a secondary key), or push hot keys into a local cache. Famous example: Twitter's "celebrity tweet" fan-out problem.
Virtual nodes / vnodes = a real machine owns many small ring positions instead of one. Used by Cassandra, DynamoDB. Smoothes load and makes adding/removing nodes cheap.

07Replication & Consensus

Replication = keeping copies of data on multiple machines. Sharding splits data; replication copies it. You usually need both.

TopologyHowUse whenRisk
Single-leader (primary/replica)All writes go to one node; replicas pull async or syncOLTP DBs (Postgres, MySQL)Replica lag → stale reads
Multi-leaderWrites accepted at many nodes; merged laterGeo-distributed editingConflicts need resolution (LWW, CRDT)
Leaderless (Dynamo-style)Writes go to N nodes; W ack, R reads mergedCassandra, DynamoDBTunable but tricky consistency
Quorum (Raft/Paxos)Majority must ack writesetcd, Spanner, CockroachDB, ZooKeeperLatency = slowest of (N/2 + 1)
Synchronous vs asynchronous replication = sync = leader waits for replicas to confirm before acking the client (safer, slower). Async = leader acks immediately, replicates in the background (faster, can lose recent writes on failover). Most cloud DBs default to one sync replica + many async followers.
Raft / Paxos (consensus algorithms) = protocols that let a group of nodes agree on the same sequence of values despite crashes. You don't implement them by hand — you use etcd, Consul, ZooKeeper, or a DB built on them (Spanner, CockroachDB). Senior expectation: know the names, the majority-quorum rule, and that they cost at least one round trip.
Quorum (W + R > N) = in a leaderless system with replication factor N, if writes acknowledge from W nodes and reads ask R nodes, then W+R>N guarantees at least one node sees both → no stale reads. Typical: N=3, W=2, R=2.
Read replica = a follower used for serving reads to take load off the primary. Replica lag means reads may be slightly stale. Don't read your own writes from a replica unless you handle it.

Delivery semantics
  • At-most-once — fire and forget; can lose messages. Cheap.
  • At-least-once — guarantees delivery, may deliver twice. Default for most systems. Forces the consumer to be idempotent.
  • Exactly-once — every message processed exactly once. Real systems achieve "effectively once" via at-least-once + idempotency keys. Kafka has transactional producers/consumers for end-to-end exactly-once within Kafka.
Dead-letter queue (DLQ) = a side queue where messages go after N failed processing attempts. Lets the rest of the pipeline keep moving while a human investigates the poison messages.
Consumer group = a set of consumers cooperating to read a topic; Kafka assigns each partition to exactly one consumer in the group. Add more consumers → process faster, up to the partition count.
Event sourcing = storing the sequence of events as the source of truth, rebuilding state by replaying them. Kafka is a popular backing store. Pairs well with CQRS (Command-Query Responsibility Segregation) for read models.

09Search & Analytics Engines

Elasticsearch / OpenSearch

Inverted-index full-text search, fuzzy match, faceting, aggregations. Best for logs, products, documents. Sharded by index; query fan-out across shards.

ClickHouse

Columnar OLAP database. Crushes group-by/aggregation queries on billions of rows. Used by Yandex, Cloudflare, Uber. Great for log analytics and time-series.

Druid / Pinot

Real-time analytics — ingest from Kafka, query in <1s. Used for dashboards over event streams.

BigQuery / Snowflake / Redshift

Cloud data warehouses. Decoupled storage + compute, SQL, petabyte scale. Used for analytics, ML feature engineering, BI.

Vector databases

Pinecone, Weaviate, Milvus, pgvector. Store embeddings, do approximate nearest-neighbor search. Used for semantic search, RAG, recommendations.

Trie / Aho-Corasick / FST

In-memory prefix or pattern structures. Used for typeahead, URL routing, dictionary matching.

Inverted index = a map from term → list of documents containing it. The core data structure of Elasticsearch. Lookups are O(1) per term plus list-merge.
Approximate Nearest Neighbor (ANN) = "find the K most similar vectors to this one, fast". Exact NN is O(N); ANN algorithms like HNSW, IVF give sub-linear time with controlled recall. The backbone of modern semantic search.

10Object / Blob Storage & CDN

Object storage

S3, GCS, Azure Blob, R2. Practically infinite, durable (11 nines), cheap (~$0.02/GB/mo). Latency 50–200 ms per object — not a database. Best for large immutable blobs: images, video, ML datasets, log archives.

Block storage

EBS, GCE PD. Looks like a disk to a VM. Used for DB data files. Not shared.

File storage

EFS, NFS, Filestore. Shared POSIX filesystem across VMs. Pricier; rarely the right answer.

CDN

Cloudflare, Fastly, Akamai, CloudFront. Hundreds of PoPs worldwide caching static and (configurably) dynamic responses near users. Honors Cache-Control. Critical for low-latency global apps.

Pre-signed URL = a temporary URL generated by your backend that grants direct, time-limited access to an S3 object. Lets clients upload/download directly to/from S3, bypassing your servers. Massive bandwidth saver.
Multipart upload = splitting a large upload into chunks uploaded in parallel, then assembled server-side. Required for objects > 5 GB on S3; recommended for anything > 100 MB.
Cache-Control headers = the HTTP headers (max-age, s-maxage, stale-while-revalidate, private, no-store) that tell browsers and CDNs how to cache a response. Knowing these by name is table-stakes for a senior front-end / platform engineer.

11Consistency Models & CAP / PACELC

Consistency = which value a reader sees after a writer commits. Stronger consistency is more expensive (latency, availability). Pick deliberately.

ModelGuaranteeExample
Strict / linearizableEvery read sees the latest committed write, instantly, globallyetcd, Spanner reads, single-node Postgres
SequentialAll clients see operations in the same orderZooKeeper
CausalCause-and-effect preserved; concurrent ops may differ across clientsCosmos DB session, Git
Read-your-writesYou see your own writes immediately; others lagMany session-mode DBs
EventualReplicas converge "eventually" with no time boundDNS, S3 (historically), Cassandra default
CAP theorem = under a network partition, a distributed system must choose between Consistency (refuse reads/writes that could be wrong) and Availability (answer with possibly stale data). CP example: etcd refuses writes on minority side. AP example: Cassandra serves whatever each side has. Note: this only matters during a partition.
PACELC = the better mental model. During Partitions: choose A or C. Else: choose Latency or Consistency. Most real-world tradeoffs are about latency vs consistency in normal operation, not partition behavior.
ACID vs BASE = ACID (Atomicity, Consistency, Isolation, Durability) — the SQL transaction guarantees. BASE (Basically Available, Soft state, Eventual consistency) — the NoSQL alternative. Most modern systems sit on a spectrum, not a binary.
Isolation levels = SQL ladder of how aggressively concurrent transactions can see each other's effects. Read uncommitted → Read committed → Repeatable read → Snapshot → Serializable. Postgres default is Read committed; you usually want Snapshot/Repeatable read for money flows.

12Distributed Transactions & Sagas

A transaction across two services (e.g., debit funds in Service A + decrement inventory in Service B) cannot use a single DB transaction. Three real options:

PatternHow it worksUse when
2-phase commit (2PC)Coordinator asks all participants "prepare?", then "commit" or "abort"Same DB family, trusted network. Avoid across teams/services — blocks on coordinator crash.
Saga (orchestrated)Central orchestrator calls services in sequence; each step has a compensating action on failureMoney flows, order pipelines, multi-service workflows. Most common in real systems.
Saga (choreographed)Each service emits events; others react; failure events trigger compensationsLoosely coupled teams; harder to reason about end-to-end
Outbox patternService writes business row + outbox row in one local DB txn; a relay publishes outbox rows to KafkaGuarantees "I committed the row → I will publish the event" without 2PC. Pairs with sagas.
Try / Confirm / Cancel (TCC)Each service exposes Reserve → Commit/RollbackPayments, ticket booking, share purchase
Compensating action = the business-level undo for a step that already succeeded. Not the same as a rollback — the side effect already happened (an email was sent, a hold was placed). The compensation explicitly counteracts it (send a cancellation email, release the hold).
Outbox pattern = avoids the dual-write problem ("how do I atomically write to DB and publish to Kafka?"). Trick: only write to the DB, including an outbox table; a CDC relay (Debezium) tails the DB log and publishes. Now there's only one source of truth.

13Idempotency & Exactly-Once

In any distributed system, retries are inevitable: timeouts, network blips, mobile OS retries. Without idempotency, retries cause double-charges, double-orders, double-everything.

The pattern. Client generates a UUID per logical operation — the idempotency key. Server stores (idemKey → result) with a unique index. On a duplicate, return the cached result instead of executing again. Stripe, AWS, PayPal, every serious payments API does this.
Where to enforce idempotency
  • HTTP layer — middleware that captures Idempotency-Key header, returns cached response.
  • DB layer — unique constraint on (user_id, idemKey).
  • Consumer layer — Kafka consumers store last-processed offset; re-deliveries are skipped.
Effectively-once delivery = at-least-once delivery + idempotent processing. From the user's perspective, indistinguishable from exactly-once. The honest framing of every "exactly-once" claim.
Optimistic concurrency control (OCC) = read with a version number; write only if version unchanged; retry on conflict. Cheap when conflicts are rare. Used by ETags in HTTP, version columns in DBs.

Token bucket vs Leaky bucket = both rate-limit, slightly different shape. Token bucket allows bursts up to capacity then steady refill (used by AWS API limits). Leaky bucket smooths into a constant outflow (used as a fair admitter, e.g., waiting rooms).
Liveness vs Readiness probe = liveness = "is the process alive?" (restart if no). Readiness = "is it ready to take traffic?" (remove from LB if no). You need both; a pod can be alive but warming up.

15Scaling Patterns

  • Vertical scaling — bigger box. Easy until you can't buy a bigger box. Limited by single-machine RAM/CPU.
  • Horizontal scaling — more boxes. Hard mode: state. Easy mode: stateless services behind an LB.
  • Stateless services — every request can go to any pod. Restart-friendly. Push state out to DB, cache, queue.
  • Auto-scaling — HPA (Kubernetes) scales pod count by CPU/memory/custom metric. VPA scales pod size. Cluster-autoscaler scales nodes.
  • Read replicas — scale reads cheaply; writes still bottleneck on the primary.
  • Caching — scales reads dramatically; doesn't help writes.
  • Sharding — the only way to scale writes past one machine.
  • Async / queue offload — push slow work behind a queue; respond fast.
  • Precompute / materialize — do the expensive thing at write time, cache the result for read time. (Typeahead pre-computed top-K is the canonical example.)
  • Multi-region / geo-distribution — for latency + disaster recovery. Costs: data residency law, cross-region replication latency, conflict resolution.
Senior heuristic. Default to stateless services + sharded data store + read replicas + cache. Reach for multi-region only when you have a real reason (compliance, RTO < 1 min, >200 ms users).

16Observability — Logs, Metrics, Traces

The three pillars. Senior expectation: name them, know what each is good at, and never use one for the wrong job.

PillarWhatToolsUse for
LogsHigh-cardinality textual recordsLoki, ELK, Datadog Logs, ClickHouseDebugging specific incidents, audit trails
MetricsNumeric time-series, pre-aggregatedPrometheus, Datadog, M3Dashboards, SLO tracking, alerts
TracesOne request's path across servicesJaeger, Tempo, Datadog APM, OpenTelemetryFinding the slow service in a chain
ProfilesCPU / memory breakdown over timepprof, Pyroscope, ParcaFinding the hot function / leak
SLI / SLO / SLA = Service Level Indicator (the measurement), Objective (your internal target), Agreement (the contract you signed with a customer). Example: SLI = "% of requests < 200 ms". SLO = 99.9% monthly. SLA = 99.5% monthly with refunds below.
Error budget = the allowed unreliability in your SLO. If SLO is 99.9%, you have 43 min/month of "allowed" downtime. Use it for risky deploys; when burned, freeze features and focus on reliability.
OpenTelemetry (OTel) = the open standard for instrumenting code with metrics, logs, and traces in one library. The 2026 default — don't write vendor-specific instrumentation.

17Security Essentials

  • TLS everywhere — including inside the cluster (mTLS via service mesh).
  • Authentication vs Authorization — authn = who you are (OAuth/OIDC, JWT, SSO). authz = what you can do (RBAC, ABAC, policies via OPA/Cedar).
  • Secret management — never in env vars or git. Use Vault, AWS Secrets Manager, GCP Secret Manager. Rotate.
  • Encryption at rest — KMS-managed keys; envelope encryption for large payloads.
  • Defense in depth — WAF at the edge, rate limit, authz at gateway, authz again at service, authz again at DB.
  • OWASP Top 10 awareness — SQLi, XSS, SSRF, broken access control, insecure deserialization.
  • PII / GDPR — minimize collection, encrypt, support deletion ("right to be forgotten"), classify data.
  • Audit logs — append-only, immutable, structured. Money/regulated systems need this.
JWT (JSON Web Token) = a base64-encoded, signed JSON blob that carries identity claims. Stateless — server doesn't need a session store. Watch out: revocation is hard; short TTLs + refresh tokens.
OAuth 2.0 / OIDC = OAuth = delegated authorization (let app X access my data on Y). OIDC = authentication layer on top of OAuth (sign in with Google/Apple/etc). "Login with X" buttons are OIDC.

18Deployment, Feature Flags & A/B

  • Blue/green — deploy new version alongside old; switch LB pointer atomically. Easy rollback.
  • Canary — 1% → 10% → 50% → 100%. Watch metrics at each step.
  • Rolling — replace pods N at a time. Kubernetes default.
  • Feature flags — code-deploy and feature-release decoupled. Toggle on/off without redeploying. Tools: LaunchDarkly, Statsig, Unleash, OpenFeature.
  • A/B tests — feature flags + metrics. Randomly assign users to variants, measure a business metric, decide statistically.
  • Shadow / dark traffic — send copies of prod traffic to the new version but discard responses. Validate behavior under real load.
  • Migrations — expand → migrate → contract. Add new column, dual-write, backfill, switch reads, drop old. Never break-then-fix.

19How to Use This in a 45-min Interview

  1. Clarify (5 min). Functional + non-functional requirements. Pin down: scale (DAU, QPS), consistency vs availability, latency SLO, data lifetime, read/write ratio.
  2. Capacity math (5 min). Use the cheat sheet in §1. Always carry units, always state B vs b, always sanity-check with the "1 day = 10⁵ s" trick.
  3. API sketch (3 min). 3–5 endpoints. POST /things, GET /things/:id, etc. Don't dwell.
  4. High-level architecture (10 min). Draw 5–8 boxes max. Separate read path from write path. Name every tech choice (Postgres, Redis, Kafka, S3 — not "the database").
  5. Data model + sharding (5 min). Pick a shard key; justify it; mention hot-shard mitigation.
  6. Deep dive on 1–2 hard parts (10 min). The interviewer will steer here. Pick the most interesting bottleneck and walk through trade-offs (saga vs 2PC, batch vs stream, etc).
  7. Failure modes + reliability (5 min). Pick the top 3 things that will go wrong; name the mitigation (RF=3, circuit breaker, retries with idempotency).
  8. Wrap with a 30-second pitch. Recap the architecture in one paragraph. Mention 2–3 things you'd revisit "with more time."
Things that separate senior from staff.
  • Name specific technologies, not categories ("Postgres with a logical replica" not "a database").
  • Always state the consistency trade-off you're making, with a why.
  • Bring up operational concerns unprompted: deploys, observability, on-call.
  • Quote real numbers: "Redis does ~150k ops/s per instance, so..."
  • Know when not to use the fancy thing. "We don't need Kafka here, an SQS DLQ is enough."
  • Talk about migration and rollback, not just the green-field design.
