Design Razorpay

A payment gateway that authorizes, captures, and settles money between buyers, merchants, and banks — correctly, exactly once, and within PCI-DSS, even when networks fail and clients retry.

01Problem & Requirements

Goal. Sit between a merchant and the banking networks (Visa/Mastercard/UPI/netbanking), accept a payment request, get it authorized and captured, and settle the money to the merchant — with correctness guarantees money systems demand.

Functional

  • Create an order; accept a payment via card / UPI / netbanking / wallet.
  • Authorize (hold funds) then capture (move funds); support refunds.
  • Settle (payout) collected money to the merchant's bank (T+1/T+2).
  • Webhooks to notify merchants of status changes.
  • Fraud / risk checks before authorization.

Non-Functional

PropertyTarget
CorrectnessNever double-charge, never lose money — exactly-once on money moves
ConsistencyStrong (ACID) on the ledger; money is not eventually consistent
Availability99.99%+ on the authorize path
Security/CompliancePCI-DSS; raw card data (PAN) never stored in app DBs
AuditabilityEvery state change journaled & reconstructable

02Back-of-Envelope Estimation

Assumptions. ~5M transactions/day · avg ticket ₹1,000 · read:write ≈ 10:1 (status checks, dashboards) · each txn writes several ledger + state rows.

Write (payment) QPS

txns/day = 5,000,000
seconds/day ≈ 86,400 ≈ 1e5
write QPS = 5,000,000 / 86,400 ≈ ~58 txns/sec
peak (festival/sale, ×10) ≈ ~580 txns/sec

Read QPS (status, dashboard, webhooks)

10:1 read:write ⇒ ~580 reads/sec average
peak ≈ ~5,800 reads/sec

Ledger write amplification

1 payment ⇒ ≥2 ledger legs (debit + credit) + state rows
≈ 5M × ~6 rows = ~30M rows/day
≈ ~350 row-writes/sec avg → must shard the ledger by merchant/account

Money throughput & storage

GMV/day ≈ 5M × ₹1,000 = ₹500 crore/day processed
ledger entry ≈ ~300 bytes ⇒ 30M × 300B ≈ ~9 GB/day
per year ≈ ~3.3 TB/year ledger (append-only, never deleted → archive cold)
Key insight. QPS is modest (~58/sec). The hard part isn't scale — it's correctness under retries and failures: exactly-once money movement, ACID ledger, idempotency, and reconciliation with banks.

03Payment Lifecycle

Two-phase money move. Authorize reserves funds on the buyer's card/account; Capture actually moves them. Splitting the two lets you verify goods/risk before taking money, and supports holds (e.g. hotels).
 created ──▶ authorized ──▶ captured ──▶ settled
    │            │              │
    │            ▼              ▼
    └────▶  auth_failed     refunded
  • created — order exists, no money moved.
  • authorized — bank approved & reserved funds.
  • captured — funds taken from buyer, owed to merchant.
  • settled — paid out to merchant bank (T+1/T+2 batch).
  • refunded — reverse a captured payment (full/partial).
The Payment Service is a state machine: each transition is validated, journaled to the ledger, and is idempotent, so a retried request never advances the state twice.

04API Design

// 1. Merchant creates an order (server-to-server)
POST /v1/orders
Idempotency-Key: "ord-9f3a..."
{ "amount": 100000, "currency": "INR", "receipt": "rcpt#42" }
→ { "id": "order_abc", "status": "created" }

// 2. Buyer pays — card is tokenized client-side, raw PAN never hits our server
POST /v1/payments
Idempotency-Key: "pay-7c1b..."
{ "order_id": "order_abc", "method": "card",
  "token": "tok_xyz" }
→ { "id": "pay_123", "status": "authorized" }

// 3. Capture (auto or delayed)
POST /v1/payments/pay_123/capture { "amount": 100000 }

// 4. Refund
POST /v1/payments/pay_123/refund { "amount": 50000 }

// 5. Status (read-heavy)
GET  /v1/payments/pay_123

// 6. Webhook to merchant (we POST to them, signed + retried)
POST https://merchant/webhook
X-Razorpay-Signature: "hmac-sha256(body, secret)"
{ "event": "payment.captured", "payload": { ... } }
  • Idempotency-Key on every money-moving POST → safe retries.
  • Token, not PAN — card data is tokenized in a PCI vault; app servers see only a token.
  • Signed webhooks let merchants verify authenticity; treat delivery as at-least-once.

06Idempotency & Exactly-Once

Idempotency key. A client-supplied unique id per logical operation. The server stores (key → result); a repeat with the same key returns the stored result instead of charging again.
  1. Client sends Idempotency-Key with the charge request.
  2. Server INSERTs the key in a unique-constrained table before calling the bank.
  3. If the insert wins → process, store result against the key.
  4. If the insert hits a duplicate → return the previously stored response (no second charge).
Why it's everything here. Networks time out and clients retry. Without idempotency a single timeout becomes a double charge. Combined with the bank's own request-id, this gives exactly-once money movement.
Distributed gotcha. "Exactly-once delivery" doesn't exist on the wire — you get at-least-once delivery + idempotent processing = effectively-once. Design every money operation to be replay-safe.

07Double-Entry Ledger

Double-entry. Every money movement is recorded as equal debits and credits across accounts, so the books always balance (Σ debits = Σ credits). This is how you make money provably correct.
-- append-only, immutable journal
ledger_entries(
  id, txn_id, account_id, direction,   -- debit | credit
  amount, currency, created_at)

-- example: capture ₹1000 from buyer to merchant (minus fee)
txn T1:
  DEBIT  buyer_settlement_account   1000
  CREDIT merchant_payable_account    980
  CREDIT razorpay_fee_account         20
  -- Σ debit (1000) == Σ credit (1000) ✓
  • Append-only: you never update/delete a ledger row — corrections are new compensating entries.
  • ACID + sharded by account/merchant; each txn is a single atomic transaction.
  • Balances are derived by summing entries (often with periodic snapshots for speed).
  • Gives you audit, reconciliation, and dispute resolution for free.

08Tokenization & PCI-DSS

  • Raw card number (PAN) never touches app servers, logs, or analytics DBs.
  • Card data is captured in an iframe / SDK and sent straight to a PCI-DSS vault, which returns a token.
  • The vault is HSM-backed, network-isolated, and the only system in PCI scope — shrinking your compliance surface.
  • All downstream services (payments, ledger, risk) operate on the token, not the card.
  • Encryption in transit (TLS) and at rest; key rotation via the HSM.
Why tokenize. It keeps 99% of your infrastructure out of PCI scope, limits blast radius of a breach (tokens are useless outside the vault), and still lets you charge a saved card.

09Webhooks & Settlement

Webhooks

  • On each state change we POST a signed event to the merchant's URL.
  • Delivery is at-least-once with exponential backoff + retries; merchants must be idempotent.
  • Merchants verify X-Razorpay-Signature = HMAC(body, secret) to trust the event.

Settlement

  • Captured funds accumulate in a merchant payable account.
  • A batch payout worker nets fees, applies holds/reserves, and initiates a bank transfer (NEFT/IMPS/RTGS) on a T+1/T+2 schedule.
  • Each payout is reconciled against bank statements; mismatches raise exceptions.
captured -> merchant_payable += amount
                 |  (T+2 batch)
                 v
   payout worker -> bank transfer -> reconcile -> settled

10Consistency & Reconciliation

  • Ledger is strongly consistent (ACID) — money is never eventually consistent.
  • Saga / orchestration across services: if a step fails, run compensating transactions (e.g. void an authorization).
  • Reconciliation jobs compare our ledger vs bank/network settlement files daily; differences become exceptions to investigate.
  • Outbox pattern publishes events atomically with the DB write so we never "charge but forget to notify".
  • Dead-letter queues capture webhooks/payouts that exhausted retries for manual handling.
Golden rule. When uncertain about a bank's outcome (timeout), do not assume — query status / reconcile. Guessing creates double-charges or lost money.

1230-Second Pitch

A payment gateway sits between merchants and the banking networks and turns a risky, retry-prone money move into a correct one. A merchant creates an order, the buyer pays with a tokenized card (raw PAN stays in a PCI-DSS vault), and a Payment Service state machine drives created → authorized → captured → settled. Every money-moving call carries an idempotency key so retries never double-charge, and every transition is journaled to an append-only double-entry ledger that always balances. Authorization is synchronous against bank connectors with fraud checks inline; settlement is async over Kafka, batching payouts to merchant banks on a T+1/T+2 schedule and reconciling against bank files. Signed, retried webhooks keep merchants in sync. The traffic is modest (~58 txns/sec) — the real engineering is exactly-once correctness, ACID money, and reconciliation when banks time out.

13Glossary

Authorize vs capture

Authorize reserves funds; capture actually moves them. Split enables holds & risk checks.

Idempotency key

Client-supplied id making a retried money request safe — returns the prior result.

Double-entry ledger

Every txn recorded as equal debits & credits so the books always balance.

Tokenization

Replace a card number with a vault token so PAN never hits app systems.

PCI-DSS

Security standard for handling card data; tokenization shrinks its scope.

Settlement

Batched payout of captured funds to the merchant's bank (T+1/T+2).

Webhook

Signed, retried server-to-server event telling the merchant a status changed.

Reconciliation

Daily compare of our ledger vs bank files to catch and resolve mismatches.

Saga / compensation

On failure, run reversing transactions (e.g. void an auth) to keep money correct.

Outbox pattern

Write the event in the same DB txn as the state change so notifications never get lost.
