00Problem & Requirements

Wise wants to raise £100 million by selling 10 million shares at £10 each (fixed). Customers log into the app, see the offer, and buy N shares using funds in their Wise account. Sale is open for 1 week. We expect 5 million concurrent users. The Balance Service (source of truth for everyone's account funds) is shared with the rest of the company — payments, withdrawals, transfers — and must not be degraded by this event.

Functional Requirements

  1. User can see remaining shares + their Wise balance.
  2. User can submit "buy N shares" within the 7-day window.
  3. Each share sold to exactly one buyer; no oversell, no undersell.
  4. Funds are debited from the user's Wise balance via the Balance Service.
  5. User gets a confirmation (success / partial fill / failure).

Non-Functional Requirements

  1. 5M concurrent users peaking when sale opens.
  2. Balance Service must not degrade — other Wise products keep working.
  3. Strong consistency on inventory — never sell the 10,000,001st share.
  4. Strong consistency on funds — never debit a customer twice.
  5. Fairness — buyers served in queue order, not "fastest internet wins." See define below.
  6. Auditable, regulator-friendly (this is securities).
Source of truth = the one system whose data is considered authoritative for a piece of state. Balance Service is the source of truth for "how much money does customer X have right now."
Fairness ("queue order, not fastest internet wins") = when supply is limited (10M shares, millions of buyers), the order in which buy requests are served must be based on when the user clicked Buy, not on which user happens to have the lowest network latency to our servers. Without fairness: a user in the same datacenter region as our cluster gets their TCP packet in 2 ms; a user across the ocean takes 200 ms. Both clicked Buy at the same wall-clock instant, but the close user always wins. That feels rigged — and for a regulated securities sale, it is a problem (regulators ask "why did this person get shares and that person didn't?"). Fix: a virtual waiting room assigns each user a queue position the moment they click, based on a server-issued timestamp + a small random jitter for ties. The actual buy flow then admits users strictly in queue order at a controlled rate — so latency to our servers no longer determines who wins. This is the same pattern Ticketmaster, IPO platforms, and PS5-launch retailers use.

01The Real Challenges

This isn't a "scale to 5M users" question. The interviewer is testing:

  1. Hot row / hot inventory — 10M shares is a single counter that millions try to decrement simultaneously.
  2. Two strongly-consistent systems must agree — share inventory and Balance Service. Both must either commit or roll back. This is a distributed transaction problem.
  3. Protecting a critical dependency — Balance Service is shared with payments/withdrawals. A spike here must not bleed into those.
  4. No double-spend, no oversell — even with at-least-once messaging and retries.
  5. Fairness at 5M concurrent users — no thundering herd, no first-come-server-wins randomness.
Hot row = a single database row that millions of requests want to update at the same time. Causes lock contention. The classic "Black Friday inventory counter" problem.
Distributed transaction = a transaction that must commit across two or more separate databases/services as one atomic unit. Either both happen, or neither. Notoriously hard.

02Capacity Math

Concurrent users:         5,000,000
Shares to sell:          10,000,000
Avg purchase per user:        2 shares  (so ~5M completed orders fit the supply)

Peak buy attempts/sec (first minute when sale opens):
  - Assume 30% of users try in first 5 minutes
  - 5M × 0.30 / 300 sec ≈ 5,000 buy attempts/sec

Read traffic (showing "shares left" + user balance):
  - 5M users polling every 5s = 1,000,000 QPS reads
  - Mostly served from CDN + cache

Balance Service writes during sale:
  - 5,000 debit/sec   (small relative to its normal load — but the spike is what kills it)

Data volume:
  - 5M completed orders × 1 KB = 5 GB of order records
  - 10M ledger entries (share-side) × 200 B = 2 GB

So the system handles ~5k writes/sec at peak, but the danger is concentrated at second 0 of the sale.

Key idea: we never let raw user traffic hammer the Balance Service directly. Everything goes through our service, which throttles and uses a dedicated connection pool.

Waiting room = an entry-gate queue that admits only N users per second into the actual purchase flow. Same pattern used by Ticketmaster, Shopify Flash Sales, IPO platforms.
Order ledger = an append-only record of every buy attempt and its outcome. Append-only = once written, never modified. Auditors love it; bugs can't corrupt history.

04Inventory — Atomic Share Decrement

We need to decrement shares_left for 5M people without ever going below zero. Three options:

OptionHowVerdict
Postgres row + SELECT … FOR UPDATELock the inventory row, decrement, commit❌ One row, 5k writes/sec → lock queue explodes. Latency spikes to seconds.
Redis DECRBYSingle-threaded atomic decrement✅ Single Redis can do 100k+ ops/sec. But still a single hot key.
Sharded Redis countersSplit 10M shares across 100 counters of 100k each; client picks a random shard✅✅ This is the answer. 100 counters × 10k ops/sec each = 1M ops/sec headroom.

Sharded inventory:

Redis keys:  inventory:shard:0  = 100,000
             inventory:shard:1  = 100,000
             ...
             inventory:shard:99 = 100,000   (total = 10,000,000)

Buy flow:
  1. Pick a random shard (or hash by userId for stickiness).
  2. DECRBY inventory:shard:k N   (atomic; returns the new value).
  3. If result < 0  →  INCRBY back; try next shard (up to 3 retries).
  4. After 3 fails  →  return "sold out" to user.
Sharded counter = splitting a single hot counter into many small ones to spread write load. Same trick used to track YouTube view counts, ad impression counts, etc.
DECRBY (Redis) = atomically subtract a number from a key, returning the new value. "Atomic" = no two clients ever see the same intermediate value.
Won't this oversell? No. Redis is single-threaded per key — DECRBY is atomic. The trick is that we INCRBY back if we go negative. The window where a value briefly goes to -3 is invisible to other clients because the next operation on that key is serialized.

05Protecting Balance Service (Bulkhead Pattern)

The Balance Service is shared. Payments, withdrawals, transfers all use it. If 5k QPS of share-purchase debits floods it, the rest of Wise goes down — and that's catastrophic.

The bulkhead pattern

"Bulkhead" comes from shipbuilding: walls inside a ship that prevent one flooded compartment from sinking the whole vessel.

Balance Service (deployed once, shared)
  ├── Connection pool A   (payments)        — 200 conns
  ├── Connection pool B   (withdrawals)     — 200 conns
  ├── Connection pool C   (share-purchase)  — 50 conns  ◄── HARD CAP
  └── Read-replica R1                       — share-purchase reads ONLY
  • Dedicated connection pool for share-purchase, capped at 50 connections. Even if our service goes wild, it can't open more.
  • Dedicated read-replica for "what's my balance?" reads. Removes 80% of our DB load from the primary.
  • Rate limiter at the edge of Balance Service: max 5k QPS from the share-purchase client, beyond that → 429 Too Many Requests.
  • Circuit breaker — if Balance Service latency exceeds 500 ms, the share-purchase service short-circuits and returns "try again in 30s" to users.
Bulkhead pattern = isolating resources (connection pools, thread pools, queues, even whole hosts) per caller so that one runaway client cannot starve every other client of the same shared dependency. Name origin: ships have internal walls called bulkheads; if one compartment floods, the wall stops water reaching the next, so the ship stays afloat. In software, the "compartment" is usually a connection pool capped at a hard limit. Concrete example for our case — the Balance Service has 450 total DB connections; we declare three pools at the Balance side: 200 for Payments, 200 for Withdrawals, 50 for Share-Purchase. If our share-purchase service goes berserk and tries to open 5,000 connections, it can only ever consume its 50; the remaining 400 are physically reserved for Payments and Withdrawals, so those flows keep working. Without a bulkhead, our spike would exhaust the whole 450-pool and take down payments too. Closely related to but distinct from a circuit breaker: a bulkhead prevents starvation (resource isolation); a circuit breaker prevents pile-on (stops calling a sick downstream).
Connection pool = a fixed-size set of pre-opened database connections, reused across requests. Limits how many concurrent queries a service can hammer the DB with.
Read replica = a read-only copy of a database that streams changes from the primary. You read from the replica to take read load off the primary, accepting milliseconds of staleness.
Circuit breaker = a wrapper that detects when a downstream service is unhealthy and stops calling it for a cooldown period. Lets the downstream recover instead of being pounded.

06Two-Phase Reserve / Saga Flow

We can't run a real distributed transaction across Redis (inventory) and Balance Service (funds). Instead, we use a saga: a sequence of local transactions with compensating actions on failure.

The buy flow (happy path)
┌─────────┐                                                          ┌──────────────┐
│  User   │                                                          │ Balance Svc  │
└────┬────┘                                                          └──────┬───────┘
     │                                                                      │
     │  1. POST /buy {shares:5, idemKey:abc}                                 │
     │ ─────────────►  Share Purchase Service                                │
     │                                                                      │
     │                 2. RESERVE inventory  (DECRBY redis)  ────────────►   │
     │                                                                      │
     │                 3. RESERVE funds      (Balance.reserve(£50,abc)) ►    │
     │                                                                      │
     │                 4. APPEND order(status=PENDING) to ledger             │
     │                                                                      │
     │                 5. COMMIT funds       (Balance.commit(abc))     ►     │
     │                                                                      │
     │                 6. APPEND order(status=COMMITTED) to ledger           │
     │                                                                      │
     │  ◄──── 7. 200 OK {shares:5, status:COMMITTED}                         │
     └─────────                                                              ┘
Compensating actions (if anything fails)
Fails at stepAction
2. Inventory reserveNothing to undo. Return "sold out".
3. Funds reserveINCRBY shares back to inventory. Return "insufficient funds".
5. Funds commit (rare — network blip)Retry. Idempotency key makes it safe. If still fails, mark order REVIEW and alert.
Saga pattern = a long-running business transaction implemented as a sequence of local transactions, each with a compensating action if a later step fails. Used instead of two-phase commit when you don't control all the databases.
Reserve / Commit (two-phase action) = first put a soft-hold on a resource ("reserve £50"), then later actually deduct it ("commit"). The Balance Service must support this — otherwise we'd need to fully debit then refund on failure, which the customer would see flicker.
Compensating action = the "undo" for a step that already succeeded, used when a later step in the saga fails. Putting shares back into inventory after a funds failure.

07Idempotency & At-Least-Once

Networks fail. Users click the buy button twice. Mobile retries hidden inside the OS happen. Without idempotency, we'd debit someone twice and oversell shares.

The pattern
  1. Client generates a UUID per buy attempt: the idempotency key.
  2. Sends it as header: Idempotency-Key: abc-123.
  3. Share Purchase Service stores (idemKey → order_id, status) in Postgres with a unique index.
  4. On a duplicate request with the same key: return the previous response instead of running the saga again.
  5. The Balance Service also accepts the same idempotency key on reserve() and commit() — so a retried commit doesn't double-debit.
Idempotent = an operation that can be applied any number of times with the same effect as applying it once. x = 5 is idempotent; x = x + 1 is not.
Idempotency key = a unique ID provided by the client so the server can recognize a duplicate of the same logical request and return the original result. Stripe, AWS, every serious payments API uses this.
At-least-once delivery = a messaging guarantee that says "your message will arrive — and possibly more than once." Almost all distributed messaging systems are at-least-once. Idempotency turns that into "effectively exactly once" from the user's perspective.

08Fair Queueing for 5M Users

The moment the sale opens, 5M users hit "buy" within seconds. If we let them all through, three bad things happen:

  • Inventory shards still get hammered (even with 100 shards, 5M concurrent is a lot).
  • Balance Service spikes.
  • Whoever has the fastest internet/closest data center "wins" — not fair.
Virtual waiting room
1. User clicks "Buy".
2. They get a token + position in a Redis sorted set:
   ZADD wait_queue {timestamp_with_jitter} {userId}
3. Their browser polls /position every 5s and shows: "You are #4,231,109 in line. Est. wait: 12 min."
4. A leaky-bucket admitter pulls N users/sec off the front:
   - rate is set so that downstream (inventory + balance) is at 60% capacity
   - typical rate: 2,000 admits/sec
5. Admitted user gets a signed short-lived token → uses it to call /buy.
6. After 10 minutes of inactivity → kicked out.
Sorted set (Redis) = a Redis data type that keeps members ordered by a score. Score = signup timestamp = position in queue.
Leaky bucket = a rate-limiting model: requests fill a bucket, leak out at a fixed rate. If the bucket overflows, new requests wait. Used here as a smooth admitter that never lets downstream get overloaded.
Thundering herd = many clients waking up and hammering a service at once. What happens when a sale opens with no waiting room. Fix: jittered timestamps, leaky bucket.

09Anti-Fraud & Kill Switches

  • Per-user cap — max 1,000 shares per customer to prevent whales gobbling up everything.
  • KYC gate — only customers with verified accounts older than 30 days can participate.
  • Bot detection — CAPTCHA / device fingerprinting at the waiting-room entry.
  • Velocity checks — same device buying for 100 accounts → flag.
  • Kill switch — feature flag that halts purchases instantly if anything looks wrong. Required for securities regulation.
  • Read-only mode — separate flag that disables writes but keeps the dashboard up.
Feature flag (kill switch) = a runtime toggle that turns a feature on/off without redeploying. Crucial for regulated systems — auditors require a "stop everything now" button.
KYC (Know Your Customer) = identity verification required for financial products. Wise already runs KYC on every account; we just check the flag.

12Interview Talking Points

  1. Lead with the two real problems: hot inventory + protecting a shared dependency. Not "scale 5M users."
  2. Sharded Redis counters for the inventory hot row.
  3. Saga + reserve/commit instead of trying to run a real distributed transaction.
  4. Bulkhead pattern protecting Balance Service: dedicated pool + read replica + rate limit + circuit breaker.
  5. Idempotency keys end-to-end (client → our service → Balance).
  6. Waiting room with leaky-bucket admitter for fairness and to keep downstream at 60% capacity.
  7. Append-only order ledger with Kafka for audit and replay.
  8. Kill switch + feature flags — it's securities, regulators demand it.
  9. Mention per-user cap for fairness — it's a domain concern but also a load mitigation.
  10. Trade-off to volunteer: "I'm using saga, not 2PC, because I don't own Balance Service's storage. The price is eventual atomicity — a customer can briefly see 'pending' state."

1330-Second Pitch

The Pitch

Two real problems: a 10-million-share hot inventory row and a shared Balance Service that must not degrade.

I'd put a virtual waiting room in front using a Redis sorted set with a leaky-bucket admitter that lets ~2k users/sec into the actual buy flow — keeps downstream at 60% capacity and gives us fairness.

For inventory, I'd shard the counter across 100 Redis keys of 100k shares each. DECRBY is atomic; if a shard goes negative I INCRBY back and try another. 1M ops/sec headroom.

For funds, the Balance Service stays shared, but my service uses a dedicated connection pool, dedicated read replica, rate limiter, and circuit breaker — classic bulkhead. Other Wise products can't see us spike.

The buy itself is a saga: reserve inventory, reserve funds, append a PENDING order, commit funds, append a COMMITTED order. Each step has a compensating action. Every call carries an idempotency key so retries don't double-debit.

Append-only order ledger for audit; kill switch for regulatory halt; per-user cap and KYC for fairness.

Strong consistency on inventory and funds; eventual consistency on reporting. Sale stays up even if reporting lags.

∞Glossary

Append-only ledger

A table where rows are only inserted, never updated or deleted. Great for auditability — history can't be rewritten.

At-least-once delivery

"Message will arrive — possibly more than once." Pair with idempotency to get "effectively exactly once."

Bulkhead pattern

Isolate resources per caller so one runaway client can't starve the others. From shipbuilding.

Circuit breaker

Wrapper that detects when a downstream is sick and stops calling it for a cooldown. Lets it recover.

Compensating action

The "undo" of a saga step, run if a later step fails.

Connection pool

Fixed-size set of pre-opened DB connections reused across requests. Hard cap on concurrent queries.

DECRBY (Redis)

Atomic subtract from a key, returns new value. Foundation of Redis-based counters.

Distributed transaction

One atomic unit of work that spans two or more separate systems. Hard. We usually avoid with sagas.

Feature flag / Kill switch

Runtime toggle to enable/disable a feature without redeploy. Mandatory for regulated systems.

Hot row / Hot key

A single record everyone wants to update at the same time. Lock contention nightmare.

Idempotent

Operation safe to repeat — repeating it has the same effect as doing it once.

Idempotency key

Client-generated UUID per logical request, lets server detect and short-circuit duplicates.

KYC

Know Your Customer — identity verification required for financial products by regulation.

Leaky bucket

Rate-limit model: requests fill a bucket that drains at a fixed rate. Smooths spikes.

Order ledger

Append-only record of every order and its outcome. Auditable, replayable, can't be corrupted.

Read replica

Read-only copy of a DB streaming from primary. Use to offload read load.

Reserve / Commit

Two-step API: first soft-hold a resource, then later finalize. Foundation of safe sagas across services.

Saga pattern

Long-running transaction as a sequence of local transactions, each with a compensating action.

Sharded counter

Splitting a hot counter into many small ones to spread write load.

Sorted set (Redis)

Members ordered by score. Used for queues and rankings.

Source of truth

The one authoritative system for a piece of state. Balance Service is the source of truth for funds.

Thundering herd

Many clients hitting a service simultaneously (e.g., sale opens). Mitigate with waiting rooms + jitter.

Two-phase commit (2PC)

Real distributed transaction protocol. Locks resources across systems. Avoided here — we use saga instead.

Waiting room

Entry-gate queue admitting users into the real flow at a controlled rate. Fairness + load shielding.
