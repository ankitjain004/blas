01Problem & Requirements

Goal. Let users store files in the cloud and keep them in sync across all their devices, with sharing, versioning, and offline support — durably and efficiently.

Functional

  • Upload / download files; create folders.
  • Automatically sync changes across a user's devices.
  • Share files/folders with other users (permissions).
  • File versioning and restore.
  • Offline edits that reconcile when back online.

Non-Functional

Durability11 nines (no data loss) on object storage
Availability99.99%
Sync latencyChanges visible on other devices in seconds

02Back-of-Envelope Estimation

Given assumptions. 50M signed-up users · 10M DAU · 10 GB free space/user · 2 file uploads/user/day · avg file size 500 KB · 1:1 read:write ratio.

Write (upload) QPS

uploads/day = 10M DAU × 2 files = 20M uploads/day
seconds/day ≈ 86,400 ≈ 1e5
write QPS = 20,000,000 / 86,400 ≈ ~230 writes/sec
peak (×5 for bursts) ≈ ~1,150 writes/sec

Read (download) QPS

1:1 read:write ⇒ read QPS ≈ write QPS ≈ ~230 reads/sec
peak ≈ ~1,150 reads/sec
total ≈ ~460 QPS average, ~2,300 QPS peak

Ingest bandwidth

data/day = 20M files × 500 KB = 10,000,000,000 KB/day = 10 TB/day uploaded
upload bandwidth = 10 TB / 86,400 s ≈ 116 MB/s ≈ ~0.93 Gbps
with 1:1 read ⇒ download ≈ another ~0.93 Gbps → ~1.9 Gbps total

Storage growth

new data/day = 10 TB/day
per year = 10 TB × 365 ≈ ~3.65 PB/year (raw)
× 3 replicas (durability) ≈ ~11 PB/year stored
dedup typically saves 30–50% → effective ~5–7 PB/year

Allocated vs. used capacity

allocated = 50M users × 10 GB = 500 PB (free-tier ceiling)
but actual usage is a small fraction — provision for real data (PB/year growth), not the 500 PB ceiling
Key insight. The 500 PB "allocated" number is a red herring — almost no one fills 10 GB. Plan storage around actual ingest (~10 TB/day) and let quota be a logical limit, not pre-allocated disk.

03API Design

Uploads are chunked and go through a metadata handshake so only missing blocks are sent.

// 1. Begin an upload \u2014 client sends file metadata + block hashes
POST /files/upload/start
{ "path": "/docs/report.pdf", "size": 524288,
  "blocks": ["sha256:ab1..", "sha256:cd2.."] }
// server replies with only the blocks it does NOT already have
→ { "uploadId": "u-123", "missingBlocks": ["sha256:cd2.."] }

// 2. Upload each missing block (often a presigned URL straight to blob store)
PUT /blocks/{hash}        (binary body)

// 3. Commit \u2014 finalize the file version from the block list
POST /files/upload/commit { "uploadId": "u-123" }

// Download / list / share
GET  /files/{id}            // returns block list + presigned GET URLs
GET  /files?path=/docs       // list a folder
POST /files/{id}/share      { "user": "bob", "role": "editor" }
  • Hash-first handshake means unchanged blocks are never re-uploaded (delta sync).
  • Presigned URLs let clients transfer bytes directly to/from blob storage, bypassing app servers.
  • Commit creates an immutable new version pointing at a block list.

05Chunking, Dedup & Delta Sync

Block-level sync. Files are split into fixed (or content-defined) chunks (e.g. 4 MB). Each chunk is hashed; the hash is its identity. Only chunks whose hash the server hasn't seen get uploaded.
  1. Client splits the file into chunks and computes each chunk's sha256.
  2. Sends the hash list; server returns which hashes are missing.
  3. Client uploads only missing chunks → delta sync (edit 1 page of a doc, upload 1 block, not the whole file).
  4. Dedup: a chunk shared by many users/files is stored once, referenced by many manifests.
Why it matters for our numbers. Delta sync + dedup is what turns 11 PB/year of replicated raw into ~5–7 PB effective, and slashes the ~0.93 Gbps upload bandwidth for edits of existing files.

06Metadata & Storage Model

Two very different stores: tiny structured metadata vs huge unstructured blocks.

-- files: the namespace + current version pointer
files(id, owner_id, path, name, is_folder, current_version_id, updated_at)

-- versions: immutable snapshots (enables history/restore)
versions(id, file_id, size, created_at, block_manifest_id)

-- blocks: content-addressed chunks (dedup key = hash)
blocks(hash PRIMARY KEY, size, storage_url, refcount)

-- manifest: ordered block list making up a version
manifest(version_id, seq, block_hash)

-- sharing
acl(file_id, user_id, role)   -- viewer / editor / owner
DataStoreWhy
Metadata (files, versions, ACL)Relational / sharded SQL (or Spanner)Strong consistency, transactions, relationships
Block contentObject storage (S3/GCS)Cheap, 11-nines durable, infinitely scalable
Hot block indexRedis / KVFast "do we have this hash?" lookups

07Sync & Notification Service

  • Each device holds a long-poll / WebSocket connection to the Notification Service.
  • On commit, the Metadata Service publishes a file-changed event (Kafka) for the owner + shared users.
  • Notification Service pushes "namespace changed since cursor X" to the relevant online devices.
  • Devices then pull the new metadata and download only the missing blocks.
  • Offline devices catch up on reconnect using a per-device sync cursor (last seen version).
Device A commit -> Metadata Svc -> Kafka(file-changed) -> Notification Svc
                                                              |
                                            push to Device B, C (online)
                                                              |
                                  B/C pull metadata diff -> fetch missing blocks

08Consistency & Conflicts

  • Metadata is strongly consistent — a commit either fully applies or not (transaction). This avoids "phantom" files.
  • Blocks are immutable & content-addressed, so they can be eventually consistent / cached freely.
  • Conflict (two devices edit offline): keep both as versions, or create a "file (conflicted copy)" — never silently overwrite.
  • Versioning makes restore trivial: point current_version_id back at an older snapshot.
Don't lose data. When in doubt, branch into a conflicted copy. Drive's golden rule: a sync system must never destroy a user's edit.

09Durability & Reliability

  • 3x replication / erasure coding in object storage → 11 nines durability.
  • Multi-region replication for disaster recovery.
  • Tiered storage: hot recent blocks on fast tiers, cold/old versions on cheap archival (Glacier-class).
  • Refcount-based GC: a block is deleted only when no manifest references it.
  • Idempotent commits keyed by uploadId so retries don't duplicate versions.

1130-Second Pitch

Google Drive splits every file into content-addressed chunks; clients hash blocks and upload only the ones the server is missing, so edits cost a few KB, not the whole file. A strongly-consistent Metadata Service owns the namespace, immutable versions, block manifests, and ACLs, while huge object storage holds the dedup'd blocks at 11-nines durability behind a CDN. A Notification Service over Kafka + WebSockets pushes change events so a user's other devices pull the metadata diff and fetch missing blocks within seconds; offline devices reconcile via a sync cursor, and conflicts become versioned copies so no edit is ever lost. For 10M DAU at 2 uploads/day that's ~230 writes/sec and ~10 TB/day ingest — modest QPS, storage-and-sync-heavy, which is why dedup and delta sync are the core levers.

12Glossary

Delta sync

Upload/download only the chunks that changed, not the whole file.

Content-addressed

A block's identity is its hash, so identical content is stored once.

Dedup

Storing one physical copy of a block referenced by many files/users.

Manifest

Ordered list of block hashes that reconstruct a file version.

Sync cursor

Per-device marker of the last version seen, used to catch up.

Presigned URL

Time-limited URL letting a client read/write blob storage directly.

Refcount GC

Delete a block only when no manifest references it anymore.

Conflicted copy

A second version kept when two offline edits collide, so nothing is lost.
