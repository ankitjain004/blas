Design a Price Search Service

Aggregate prices for a product from many vendor APIs and return them to the front-end — while honoring 1-hour data freshness, minimizing expensive vendor calls, and scaling to 100+ vendors.

Vendor fan-out · 1h TTL cache · cost-aware calls · async refresh

01Problem & Requirements

Goal. A customer searches for a product (e.g. productId = 1, "Travel Insurance"). The service queries multiple vendor APIs (A, B, C…), aggregates their prices, and returns a unified list to the front-end.

Functional

  • GET /search/product/{id} returns the latest known price from every vendor for that product.
  • Each result carries the vendor, price, currency, and freshness (timestamp / age).
  • New vendors can be onboarded without redeploying core logic.

Constraints given in the brief

#ConstraintDesign Implication
1Vendor data is valid for 1 hourCache with 1h TTL; never call a vendor if a fresh value exists.
2Each vendor API call costs moneyMinimize calls — cache-first, dedupe, batch, refresh lazily.
3(Challenge) scale to 100+ vendorsAsync fan-out, bounded concurrency, per-vendor isolation, partial results.

Non-Functional

PropertyTarget
Read latency (cache hit)< 50 ms p95
Read latency (cold, partial)return cached + refresh async; never block on all vendors
Availability99.9% — degrade gracefully if some vendors are down
Vendor costMinimize redundant calls; aim < 1 call/vendor/product/hour

02The Naive Design & Why It Hurts

The teammates' first cut: an API that takes a product id, synchronously calls every vendor, writes the result to a DB, and returns it.

UI --GET /search/product/1--> SearchService
        SearchService --1/A--> Vendor A
        SearchService --1/B--> Vendor B
        SearchService --1/C--> Vendor C
        SearchService --save {A:$100,B:$120,C:$110}--> DB
UI <-- {A:$100, B:$120, C:$110} -- SearchService
Why this breaks down.
  • Slowest vendor sets your latency. Synchronous fan-out means one slow/timed-out vendor blocks the whole response.
  • Cost explosion. Every search hits every vendor, even when a price was fetched seconds ago — directly violates constraint #2.
  • No real cache. Writing to the DB after the fact isn't a cache; the next request still fans out.
  • One bad vendor fails the request. No isolation or partial results.
  • Doesn't scale to 100+ vendors. 100 synchronous calls per request is a non-starter.

03API Design

Request

GET /search/product/1

Response

{
  "productId": 1,
  "results": [
    { "vendor": "A", "price": 100, "currency": "USD", "asOf": "2026-06-04T10:00:00Z" },
    { "vendor": "B", "price": 120, "currency": "USD", "asOf": "2026-06-04T10:02:00Z" },
    { "vendor": "C", "price": 110, "currency": "USD", "asOf": "2026-06-04T09:58:00Z" }
  ],
  "stale": false
}
  • Idempotent & cacheable. A GET with no side effects; safe to put behind a CDN/edge cache with a short TTL.
  • Per-vendor freshness. asOf lets the client reason about age; stale flags a degraded (cached-while-refreshing) response.
  • Pagination/sort. For many vendors, support ?sort=price&limit=20 and stream the cheapest first.
  • Onboarding API. POST /vendors registers a vendor adapter (endpoint, auth, rate limit, cost) so new vendors are config, not code.

05Caching Strategy (1-Hour TTL)

Cache-aside with a per-vendor freshness window. Key: price:{productId}:{vendor} → {price, currency, asOf} with TTL = 1h. A product-level aggregate price:{productId} can also be cached for the hot read.
  1. Read the product aggregate from Redis. If present and every vendor entry is < 1h old → return immediately (the cheap, fast path).
  2. On miss/stale, return whatever is cached (marked stale) and enqueue a refresh for only the stale vendors — don't block the user.
  3. Workers fetch fresh prices, write to Redis (TTL 1h) and the durable DB, then update the aggregate.
Stale-while-revalidate. Serving slightly-stale cached data instantly while refreshing in the background gives fast responses and bounded staleness — ideal when the brief says data is "valid for 1 hour."
LayerRoleTTL
Edge/CDNAbsorb repeated identical readsseconds–minutes
Redis (hot)Per-vendor + aggregate prices1 hour
Price DB (jsonb)Durable source of truth, survives cache flushpersistent

06Optimizing Vendor Cost

Constraint #2 says every vendor call costs money. The whole design optimizes for fewest paid calls:

  • Cache-first. A fresh cached value means zero vendor calls — the biggest lever.
  • Request coalescing (single-flight). If 1,000 users ask for product 1 while it's stale, only one refresh task runs per vendor; the rest wait on/await that result.
  • Refresh only what expired. Refresh per-vendor, not the whole product, so a vendor refreshed 5 minutes ago isn't called again.
  • Batch endpoints. If a vendor supports multi-product queries, fetch many productIds in one paid call.
  • Proactive vs lazy. For hot products, a scheduler pre-warms just before the 1h expiry; cold products refresh lazily on demand only.
  • Per-vendor rate limits & budgets. Token-bucket per vendor to cap spend and respect their quotas.
Single-flight is the key cost saver. Without it, a cache stampede on a popular product turns one expiry into thousands of paid calls.

07Scaling to 100+ Vendors

  • Async fan-out, not sync. Publish a refresh event; a worker pool calls vendors in parallel with bounded concurrency — never 100 blocking calls in the request thread.
  • Per-vendor isolation (bulkhead). A circuit breaker + rate-limit budget is tracked per vendor, so vendor X being slow or down can't starve vendor Y — even though workers are shared.
  • Partial results. Aggregate whatever returns within a deadline; missing vendors are filled in on the next refresh and surface as stale.
  • Config-driven adapters. Vendor endpoint, auth, format, cost, and rate limit live in a registry — onboarding is data, not deploys.
  • Horizontal scale. Partition the refresh topic by vendor (or productId) so workers scale out independently; DLQ for poison messages.
Refresh event {productId:1, vendors:[stale...]}

Worker Model — pool, not 1-per-vendor

Common misconception: you do not run one worker per vendor. With 1,000 vendors that would mean 1,000 mostly-idle threads — it doesn't scale and wastes memory/connections.

Instead, run a fixed, elastic pool of workers (e.g. 20–50). They pull jobs off the queue; each worker handles whatever vendor job it grabs next. This is bounded concurrency — it decouples the number of vendors from the number of workers.

1 worker per vendorWorker pool (chosen)
100 vendors100 threads, mostly idle~30 workers, always busy
1000 vendors1000 threads — falls overadd workers, not threads-per-vendor
Resource useWasted memory + connectionsBounded, predictable
Scaling knobPinned to vendor countPool size + Kafka partitions
What stays per-vendor is policy, not threads. Two things are keyed by vendor and enforced by whichever worker runs the job:
  • Rate limit / budget — a token bucket per vendor so you never exceed Vendor A's quota or overspend, regardless of which worker calls it.
  • Circuit breaker — per-vendor breaker state. If Vendor B is down, any worker that picks a Vendor B job sees the breaker open and skips fast, so one bad vendor can't tie up the whole pool.

With 30 workers and 100 vendors, all 100 vendor-calls queue up and 30 run concurrently at any instant; as one finishes, that worker pulls the next. Partitioning the Kafka topic by vendor keeps each vendor's jobs ordered and lets you scale noisy vendors independently — workers remain a shared, elastic pool.

Graceful degradation. A search returns the cheapest available prices even if a few vendors are temporarily unavailable — partial > nothing.

09Database Choice & Data Model

Why a relational store (PostgreSQL) as the source of truth

The data is small, structured, and read-dominated — a product keyed to a handful of vendor prices. The access pattern is a point lookup by product_id. That makes a single, well-understood relational DB the durable source of truth, with Redis as the hot cache in front.

OptionFitVerdict
PostgreSQL (chosen)Point lookups by PK, JSONB for the vendor list, transactions for selective per-vendor upserts, cheap to operate✅ Durable source of truth
RedisSub-ms reads, native TTL = the 1h freshness window✅ Hot cache, not durable store
Cassandra / DynamoDBGreat if write-throughput/partitioning dominated, but here volume is modest⚠️ Overkill; use only if products explode to billions
ElasticsearchOnly if the brief becomes fuzzy text search over products⚠️ Different problem; not for price lookup
Rule of thumb: the workload is cache-served point reads with a 1-hour TTL. Redis handles the hot path; Postgres just needs to be durable and survive a cache flush — it is not the bottleneck, so don't reach for a distributed NoSQL store prematurely.

Schema decision — why both an aggregate row and a per-vendor table

The brief suggests a single product-keyed row with a JSONB blob of vendor prices. That serves the hot read in one lookup, but it can't express per-vendor freshness or selective refresh. So we keep both:

-- Aggregate (matches the brief)
CREATE TABLE product_prices (
  product_id  BIGINT PRIMARY KEY,
  results     JSONB,        -- [{vendor, price, currency, asOf}, ...]
  updated_at  TIMESTAMPTZ
);

-- Per-vendor (for selective refresh, cost & freshness)
CREATE TABLE vendor_prices (
  product_id  BIGINT,
  vendor_id   TEXT,
  price       NUMERIC,
  currency    TEXT,
  fetched_at  TIMESTAMPTZ,
  PRIMARY KEY (product_id, vendor_id)
);
  • results JSONB serves the hot read in one row lookup.
  • vendor_prices.fetched_at drives the 1h TTL decision and selective refresh.
  • Index on fetched_at lets a scheduler find soon-to-expire hot products to pre-warm.

1230-Second Pitch

The Price Search Service is a cache-first aggregator. Reads hit Redis with a 1-hour TTL and return instantly; the user request never blocks on vendors. On a miss or stale entry we serve what we have (flagged stale) and publish a refresh event. A bounded worker pool fans out to vendors asynchronously — with per-vendor circuit breakers, retries with backoff, and single-flight so a stampede on a hot product costs one call, not thousands. Fresh prices are written to Redis and a durable jsonb store. New vendors are config-driven adapters, so scaling to 100+ vendors is parallel fan-out plus per-vendor isolation, not a rewrite. The result is fast reads, minimal vendor cost, and graceful degradation when vendors fail.

13Glossary

Cache-aside

App checks cache first; on miss it loads the source and populates the cache.

Stale-while-revalidate

Serve slightly old data instantly while refreshing it in the background.

Single-flight

Collapse many concurrent identical refreshes into one underlying call.

Circuit breaker

Stop calling a failing vendor for a cooldown so you don't waste cost/time.

Bulkhead

Isolate resources per vendor so one bad vendor can't starve the others.

DLQ

Dead-letter queue holding messages that repeatedly failed processing.

TTL

Time-to-live; here, 1 hour of validity before a price must be refreshed.

Backoff + jitter

Increasing, randomized retry delays to avoid synchronized retry storms.
