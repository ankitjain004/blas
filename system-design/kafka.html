01What Kafka Is (and Isn't)

One-liner. Kafka is a distributed, append-only commit log — a durable, replayable stream of events that decouples producers from consumers. Think "a library, not a mailbox": you read a record but it stays on the shelf for others (and for you again) to re-read.
Why it exists
  • Decoupling — producers write and move on; consumers read at their own pace. Neither needs to know the other.
  • Replayability — because records are retained (not deleted on read), you can reprocess from any offset after a bug or to backfill a new consumer.
  • Throughput — sequential disk writes + zero-copy + batching make a single broker handle hundreds of MB/s.
  • Durability — records are replicated across brokers; a broker can die without data loss.
Not a traditional queue. Unlike RabbitMQ/SQS, a Kafka consumer reading a record does not remove it. Consumers track their own offset (a bookmark). This is the key mental shift.

02Architecture & Components

⬇ Download Interactive Diagram (.excalidraw)   ↗ Open in Excalidraw

Broker = a single Kafka server. A cluster is many brokers (3, 5, 100…). Each broker stores a subset of partitions and serves reads/writes. Add brokers to scale storage and throughput.
Topic = a named stream/category of records (like a table name). Producers write to a topic; consumers read from it. Example: messages.sent, orders.created.
Partition = an ordered, append-only log that a topic is split into. The unit of parallelism and the unit of ordering. More partitions → more parallel consumers.
Offset = the sequential ID (0,1,2…) of a record within a partition. Consumers commit "I've read up to offset N" → enables replay and resume.
Producer = client that writes records. Chooses the partition (by key hash or round-robin) and waits for acks (acks=all = wait for replicas).
Consumer / Consumer Group = client(s) that read records. Within a group, each partition is assigned to exactly one consumer → parallel processing with no duplication.
KRaft / ZooKeeper = cluster coordinator for metadata, leader election, and broker membership. Modern Kafka (KRaft) self-manages metadata via Raft; older Kafka used ZooKeeper.

03Partitions & Ordering

Golden rule. Ordering is guaranteed within a partition, never across partitions. To keep related events ordered, route them to the same partition via a partition key.
producer.send(key = "channel_42", value = msg)

  partition = hash(key) % numPartitions      ← same key → same partition → ordered

Topic: messages   (4 partitions)
  P0: [m0][m1][m2][m3]  ← append-only, offset increases →
  P1: [m0][m1][m2]
  P2: [m0][m1][m2][m3][m4]
  P3: [m0][m1]
ChoiceEffect
Key = channel_idAll messages of one chat stay ordered (same partition).
No key (round-robin)Max throughput, but no ordering guarantee across records.
More partitionsMore parallelism, but more open file handles, more rebalancing, and global ordering is lost.
Hot partitions. A skewed key (e.g., one celebrity user) overloads a single partition. Mitigate with a composite key or salting.

04Consumer Groups

Topic (3 partitions)        Consumer Group "delivery-workers"
  P0 ─────────────────────►  Consumer A
  P1 ─────────────────────►  Consumer B
  P2 ─────────────────────►  Consumer C

  Add a 4th consumer → it sits IDLE (only 3 partitions).
  To scale further → add partitions.
  • 1 partition → 1 consumer within a group. This caps a group's parallelism at the partition count.
  • Rebalancing — when a consumer joins/leaves, partitions are reassigned. Causes a brief pause; minimize with cooperative/incremental rebalancing.
  • Multiple groups read the same topic independently (pub/sub): each group has its own offsets.
  • Offset commit — commit after processing for at-least-once; commit before for at-most-once.

05Replication & Durability

Partition 0 (replication factor = 3)
  Broker1: LEADER     ◄── all reads/writes
  Broker2: follower   (replicates leader)
  Broker3: follower   (replicates leader)

  ISR = In-Sync Replicas (followers caught up).
  Leader dies → a follower in ISR is promoted.
  • Replication factor (RF=3 typical) = copies of each partition across brokers.
  • Leader handles all I/O; followers mirror it. Clients only talk to the leader.
  • ISR = replicas fully caught up. acks=all + min.insync.replicas=2 = a write is acked only after it's on ≥2 replicas → survives one broker loss.
  • Unclean leader election = allowing an out-of-sync replica to become leader → availability over durability (can lose data). Usually disabled.

06Delivery Semantics

GuaranteeHowTrade-off
At-most-onceCommit offset before processingFast, but may lose messages on crash
At-least-onceCommit offset after processingNo loss, but possible duplicates → need idempotent consumers
Exactly-onceIdempotent producer + transactions (transactional.id)Strongest, but more overhead; scoped to Kafka-to-Kafka
Practical default. Most systems use at-least-once + idempotency keys on the consumer (dedup on a unique message ID). Exactly-once is reserved for stream-processing pipelines (Kafka Streams).

07Retention & Compaction

  • Time/size retention — keep records for retention.ms (e.g., 7 days) or retention.bytes, then delete the oldest segments. Independent of whether they were consumed.
  • Log compaction — instead of deleting by age, keep the latest value per key. Perfect for changelog/state topics (e.g., "current profile per user").
  • Segments — partitions are split into segment files; old segments are dropped or compacted in the background.
Tiered storage = offload old segments to object storage (S3) while keeping recent data on local disk — cheap long retention without bloating brokers.

Consumer lag is the #1 health metric: lag = log-end-offset − committed-offset. Rising lag means consumers can't keep up → add consumers (up to partition count) or partitions.

10Interview Talking Points

  • "Kafka is a replayable commit log, not a mailbox — consumers track offsets, records are retained."
  • "Ordering is per-partition; I route related events with a partition key to preserve order."
  • "Parallelism is capped by partition count — one partition per consumer in a group."
  • "Durability comes from replication + ISR; acks=all with min.insync.replicas survives broker loss."
  • "I default to at-least-once + idempotent consumers; exactly-once only for Kafka-to-Kafka streams."
  • "I monitor consumer lag as the primary health signal."

11Glossary

Broker

A single Kafka server storing partitions and serving I/O.

Topic

Named stream of records; logically split into partitions.

Partition

Ordered, append-only log; unit of parallelism and ordering.

Offset

Position of a record within a partition; enables replay/resume.

Consumer Group

Set of consumers sharing partitions; 1 partition → 1 consumer.

ISR

In-Sync Replicas — followers fully caught up with the leader.

Consumer Lag

log-end-offset − committed-offset; how far behind a consumer is.

Log Compaction

Retain only the latest value per key for changelog topics.

KRaft

Kafka's Raft-based metadata mode that replaces ZooKeeper.
