01Problem & Requirements

Goal. Suggest the top people a user is likely to know and connect with, while respecting blocks, privacy settings, and quality constraints.
  • Show top-N recommendations per user (e.g., 20 cards).
  • Support actions: connect, dismiss, hide, report.
  • Update suggestions as graph changes (new friends, imports, groups, jobs, schools).
  • Avoid already-connected users and blocked users.

02Capacity Math

Notation. B is byte, b is bit, 1 B = 8 b.
  1. 100M DAU, each opens PYMK 2 times/day → 200M feed requests/day.
  2. QPS ≈ 200M / 86,400 ≈ 2,315 avg; at 10x peak ≈ 23k QPS.
  3. If each response has 20 candidates × 120 B summary payload ≈ 2.4 KB/resp.
  4. Egress at peak ≈ 23k × 2.4 KB ≈ 55 MB/s.
  5. Graph: 1B users, avg 250 edges/user (undirected) → ~125B edges.
  6. Store compressed adjacency + counters + feature store in distributed graph/kv systems.

05Online Request Flow

  1. User opens PYMK. API hits PYMK service with userId/context.
  2. Service checks precomputed Top-N cache for user segment.
  3. On miss or stale entry: generate 500-2,000 candidates from graph traversals and recall indexes.
  4. Join features (mutuals, overlap, recency, quality priors).
  5. Apply policy filter first (blocked/already connected/private accounts).
  6. Score with model (GBDT/NN), diversify, return top 20.
  7. Write impressions and actions to event stream for training feedback.

06Offline Pipeline

  • Nightly batch computes heavy graph features: Adamic-Adar, personalized PageRank-like proximity, community overlap.
  • Near-real-time stream updates fast features: new edge, dismiss, block, accept.
  • Feature store keeps offline+online parity for training/serving consistency.
  • Candidate snapshots precomputed for cold users and long-tail users.
Feature freshness tiers: hot (seconds), warm (minutes), cold (daily) to balance relevance and cost.

07Ranking Strategy

StageCountTechnique
Recall5k-20kGraph heuristics + ANN-like neighbor retrieval

Final rank100-500Rich model with calibrated score
Post-rankTop 20Diversity/fairness/novelty constraints
Important. Dismiss actions must down-rank fast. Accept actions should increase similar-candidate weights, but cap reinforcement to avoid filter bubbles.

08Privacy, Safety, Abuse

  • Hard exclusions first: blocked users, minors policy, legal restrictions.
  • Respect profile visibility and discoverability settings.
  • Hash+salt sensitive imported contacts, separate access controls.
  • Rate-limit and detect growth-hacking abuse bots.
  • Explainability snippets: "You both studied at X", "12 mutual connections".

1030-Second Pitch

People You May Know is a multi-stage recommendation system. We first generate high-recall candidates from a social graph (mostly two-hop and similarity indexes), then join online/offline features, apply strict privacy and safety filters, and rank with a model to return top-N suggestions under 150 ms. Heavy graph computations are precomputed offline, while key user actions stream in near-real-time to keep suggestions fresh. Blocks, existing connections, and policy checks are always hard constraints before ranking.

11Glossary

Adamic-Adar

Graph similarity metric weighting rare common neighbors higher than popular ones.

Two-hop neighbors

Friends-of-friends; efficient high-recall source for connection candidates.

Feature store

Central system serving consistent model features for training and online inference.

Pre-rank

Fast intermediate stage that trims candidate set before expensive final ranking.

Diversity rerank

Post-processing to avoid repetitive suggestions from the same community.

Policy filter

Non-negotiable privacy/safety gates applied before ranking outputs.
