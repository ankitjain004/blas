00Problem & Requirements

Build a "search-as-you-type" suggestion service: as the user types one keystroke at a time, return the top 5 most likely completions, drawn from global past searches.

Functional Requirements

  1. Show top 5 suggestions while user is typing.
  2. Suggestions are based on global past searches (what everyone has searched).
  3. Collect each user's completed search back into the system to improve future suggestions.

Non-Functional Requirements

  1. ~10⁹ searches per day → ~10k QPS.
  2. Each keystroke triggers a request → real traffic ≈ 100k QPS (10× typed chars).
  3. Low latency: order of milliseconds (target p99 < 100 ms).
  4. Availability > Consistency — stale suggestions are fine for a few minutes.
Typeahead = the dropdown that pops up as you type into a search box, suggesting possible completions. Examples: Google search, Amazon product search, IDE autocomplete.
QPS = "Queries Per Second" — how many requests the system handles each second. 10k QPS = ten thousand queries every second on average.
p99 latency = the response time at the 99th percentile. p99 < 100 ms means 99% of requests come back in under 100 ms.
AP vs CP = the two main flavors of distributed systems (from the CAP theorem). AP = "Availability + Partition tolerance" — the system always responds, even if data is slightly stale. CP = "Consistency + Partition tolerance" — always correct, but may refuse requests during issues. For typeahead, slightly stale is fine; downtime is not. So we pick AP.

01Capacity Math

Notation we'll use everywhere
b = bit (lowercase) · B = byte (uppercase) · 1 B = 8 b
1 KB = 1,000 B · 1 MB = 10⁶ B · 1 GB = 10⁹ B · 1 TB = 10¹² B (decimal / SI units, the standard in capacity math)
1 char = 1 B for ASCII text (UTF-8 keeps Latin letters at 1 B; emoji/CJK would be 2–4 B — we ignore for this back-of-envelope).
Step 1 — Read QPS (queries per second)
Daily searches               = 10⁹ searches/day
Seconds per day              = 24 × 3600   = 86,400 s/day ≈ 10⁵ s/day
Average completed-search QPS = 10⁹ / 10⁵   = 10⁴       = 10,000 QPS

But each KEYSTROKE fires a request (after 100 ms debounce):
  Average query length       = 15 characters     (1 char = 1 B in UTF-8 ASCII)
  Requests per finished query≈ 15 / debounce ≈ 10 network calls
  Keystroke QPS              = 10,000 × 10  = 100,000 QPS  (call it ~10⁵)
  Peak (2× headroom)         = 200,000 QPS ≈ 2 × 10⁵ QPS
Step 2 — Write QPS (learning from completed searches)
Only the FINAL submitted query is recorded (not every keystroke):
  Write QPS                  = 10,000 events/sec  = 10⁴ events/s
  Each event payload         ≈ 200 B  (query + userId + ts + metadata)
  Kafka ingress              = 10⁴ × 200 B = 2 × 10⁶ B/s = 2 MB/s
Step 3 — Storage (the trie)
Distinct queries kept        = 5 × 10⁹    (5 billion unique phrases over 30 days)
Avg phrase length            = 20 characters = 20 B
Raw phrase text              = 5×10⁹ × 20 B = 10¹¹ B = 100 GB

Trie overhead factor         ≈ 5×       (child pointers 8 B each + freq counter 8 B
                                          + top-5 list ≈ 5 × 16 B = 80 B per node)
Trie in RAM                  ≈ 100 GB × 5 = 500 GB total

Per-shard RAM (50 shards)    = 500 GB / 50 = 10 GB / shard
Plus 3 replicas for HA       = 150 trie-server processes in total
Step 4 — Network bandwidth
Suggestion response size:
  5 suggestions × ~30 B each = 150 B
  + JSON envelope, headers   ≈ 850 B
  Total response             ≈ 1 KB = 10³ B

Downstream bandwidth (to users), assuming CDN absorbs 90%:
  Origin QPS  = 200,000 × 10% = 20,000 QPS
  Origin BW   = 20,000 × 1 KB = 20 MB/s
  Edge BW     = 200,000 × 1 KB = 200 MB/s (served by CDN, not us)
Step 5 — Summary in one row
Reads   : 200,000 QPS peak  · 1 KB/response  · <30 ms p99
Writes  :  10,000 evt/s     · 200 B each     · async
Storage : 500 GB total      · 10 GB / shard  · in RAM
Net out :  20 MB/s origin   · 200 MB/s edge
Debounce = wait until the user pauses typing (e.g., 100 ms) before sending a request. Avoids firing one request per keystroke. The keyboard always fires per-key locally, but the network call only goes out after the pause — so we count ~10 calls per 15-char query, not 15.
Byte (B) vs bit (b) = 1 byte = 8 bits. Always capital B for byte, lowercase b for bit. Network speeds are often quoted in bits (Gbps = gigabits per second), storage in bytes (GB). Mixing them up gives 8× wrong answers in interviews.

02The Core Data Structure — Trie

A trie (pronounced "try") is a tree where each path from the root to a node spells out a prefix. Perfect for "find words starting with X" lookups.

Example trie holding: "cat", "car", "cap", "cab", "can"

                    root
                     │
                     c
                     │
                     a
                  ┌──┼──┬──┬──┐
                  t  r  p  b  n
                  ●  ●  ●  ●  ●   (● = end-of-word, holds frequency)

Walking "ca" lands us at the node after 'a'. To suggest, we look at all words
in the subtree below that node and pick the top 5 by frequency.
Trie (Prefix Tree) = a tree where each edge is one character; a path from root spells a prefix. Lookup of "ca" is just 2 hops, regardless of how many million words are stored. That's why typeahead loves tries.
The naive approach (and why it's too slow)

For every keystroke, we'd:

  1. Walk the trie to the node matching the typed prefix (O(prefix length), fast).
  2. DFS the entire subtree to collect all words below.
  3. Sort them by frequency, take top 5.

Step 2 is the killer — a popular prefix like "a" could have millions of words underneath. At 150k QPS, this is impossible in milliseconds.

DFS (Depth-First Search) = walk a tree by going as deep as possible before backtracking. For a 5-billion-word trie, walking even part of it is way too slow per request.

03Pre-computed Top-K per Node

The trick: at every trie node, store the top 5 completions of words passing through that node, pre-computed offline.

                    root
                     │
                     c
                     │
                     a              top5 = ["car", "can", "cat", "cap", "cab"]
                  ┌──┼──┬──┬──┐
                  t  r  p  b  n
                  ●  ●  ●  ●  ●

Now, a query is just:

  1. Walk the trie to the prefix node (O(prefix length)).
  2. Return the pre-computed top-5 list. No subtree walk needed.
Trade-off: Reads become O(prefix length) — typically <1 ms. Writes are now expensive: when a word's frequency changes, we may have to update the top-5 at every ancestor node. So writes are batched offline (next section).
Top-K = the K most popular / most relevant items. For typeahead, K = 5. We pre-rank these so the read path is dead simple.

Happy-path flow for a single keystroke
  1. User types c, then a in the search box.
  2. Browser debounces 100 ms (so we don't fire on every keystroke), then sends GET /suggest?q=ca.
  3. Request hits the nearest CDN edge (e.g., Cloudflare PoP in the user's city).
    • Cache hit (≈ 90% of the time for short prefixes like "ca") → CDN returns the cached top-5 JSON in < 10 ms. Request never touches our servers.
    • Cache miss → continue.
  4. CDN forwards to API Gateway (auth, per-IP rate limit, load balance) which picks one stateless Typeahead Service pod.
  5. Typeahead Service hashes the prefix (hash("ca") mod 50) to choose a shard, then picks a healthy replica of that shard.
  6. In parallel:
    • Trie shard does an in-memory walk: root → 'c' → 'a' (2 pointer hops) → returns pre-computed top-5. < 1 ms.
    • Redis: ZREVRANGE trending:ca 0 4 → returns top-5 trending queries for this prefix (last 2h). ~0.5 ms.
  7. Typeahead Service merges both lists (deduplicates), re-ranks with 0.7 × trie_score + 0.3 × redis_score, returns final top-5. This surfaces trending terms AND brand-new terms not yet in the trie.
  8. Response (~1 KB JSON) flows back: Typeahead → Gateway → CDN. CDN writes it into its edge cache with a 30–60 s TTL.
  9. Browser renders the dropdown. End-to-end latency typically 5–30 ms.
Why the edge cache exists (and why it's the most important box)
  • Closes the distance to the user. A user in Mumbai hitting an origin in Virginia = ~250 ms RTT before we even compute anything. A Mumbai CDN PoP = 5–10 ms RTT. Latency win without any work.
  • Absorbs the Zipf head. 0.1% of prefixes ("a", "th", "fac", "yo") get ~90% of traffic. Caching just those few thousand entries deflects 90% of QPS — we only build origin capacity for the remaining 10%.
  • Cheap to serve. CDN bandwidth is far cheaper than running 10× more trie servers.
  • Acts as a buffer during outages. If our origin has a hiccup, CDN can keep serving stale-but-good responses (stale-while-revalidate).
  • What we cache: the JSON response keyed by the prefix string + locale (e.g., en-US:ca). TTL 30–60 s is the sweet spot — short enough to pick up new trends, long enough for high hit rate.
CDN (Content Delivery Network) = a global network of edge servers (Cloudflare, Fastly, Akamai) physically close to users. Turns a 250 ms cross-country round trip into a 5 ms one.
Edge cache = the cache layer at each CDN PoP (Point of Presence). Keyed by request URL + headers; honors Cache-Control headers from the origin. For "ca", we cache the top-5 response right at the edge and never hit our backend.
API Gateway = the single entry point for clients. Handles auth, rate limiting, request validation, and routing to the right backend service.
Trie Server = a process that holds one shard of the trie entirely in RAM. Lookups are pure pointer-chasing in memory — no disk, no network — so they finish in < 1 ms.
Kafka = the durable append-only queue we use to move events between services. Each completed user search is published as one message on the queries.raw topic; Flink/Spark consume it.

05Sharding the Trie

500 GB of trie data won't fit in one server's RAM (a fat box has ~256 GB). We shard the trie across N servers.

First, what "sharding" actually means.

Sharding is a concept, not a product. It just means: split one logical dataset into many physical pieces by some key, and route each request to the piece that owns its key. The pieces can live in:
  • Databases — Postgres logical partitions, MongoDB shards, Cassandra rings, DynamoDB partition keys.
  • Caches — Redis Cluster slots, Memcached consistent-hash rings.
  • Application servers holding in-memory state — like our trie servers here, or game-server "rooms" sharded by room-id.
  • Message queues — Kafka topic partitions (we already used this in the logging design).
Same idea in every layer: function(key) → shard-id → physical node. In our typeahead case, there is no database on the read path — the trie itself lives in RAM inside the application servers, and those are what's sharded. The "database" of completed searches lives separately in Kafka / a data warehouse and is only used by the offline builder.
Two sharding strategies
StrategyHowProsCons
By first letter / prefix range'a*' → shard 1, 'b*' → shard 2, …Each request goes to exactly one shard; build can produce one trie per shard directlyUneven load — "a", "s", "t" are hot; "z", "q" are cold
By hash(prefix)For each typed prefix, compute its hash and mod by NEven load across shardsSame word's prefixes scatter across many shards → harder to compute global Top-K during build

In practice: shard by first 2 characters (~26² = 676 logical buckets), then group buckets into ~50 physical shards balancing by traffic (the hot "th*", "fa*", "yo*" buckets get their own shards; cold ones are merged). Each shard gets 3 replicas for HA and read scaling.

How a request finds its shard at runtime
  1. API Gateway / Typeahead Service has a shard map in memory: bucket → shard-id → list of replica addresses. Loaded from a config service (ZooKeeper / etcd / Consul) and watched for changes.
  2. For prefix "ca", compute bucket = "ca" → look up shard-id = 7 → list of 3 replica addresses.
  3. Pick a healthy replica (round-robin, or "nearest" based on latency).
  4. Send the request. If it times out, retry on another replica.
Shard = one slice of a larger dataset living on a different machine. Each shard owns a deterministic subset of keys. Sharding ≠ replication. Sharding splits the data; replication copies a shard for HA.
Replica = a copy of a shard, for redundancy and read scaling. 3 replicas per shard = if one dies, the other two keep serving with no data loss.
Shard map / routing table = the small piece of metadata that says "which shard owns which key". Stored in a coordination service like ZooKeeper or etcd; every Typeahead Service pod keeps a cached copy.

06Update Pipeline (batch + stream)

How does new search data get into the trie? Lambda architecture — a slow batch path that rebuilds everything, and a fast stream path that patches recent changes.

queries.raw = the name of one specific Kafka topic we created for this system. A "topic" in Kafka is just a named, append-only log file (replicated across brokers). Every time a user submits a search like "cats videos", the Search Service appends one JSON record to this topic, e.g. {"q":"cats videos","uid":123,"ts":1716...,"locale":"en-US"}. The topic retains these records for 30 days, so any downstream consumer (batch job, stream worker, audit) can replay them.
Lambda architecture = a pattern that splits data processing into two parallel pipelines: a batch layer (slow but accurate, recomputes everything from scratch on a schedule) and a speed/stream layer (fast but approximate, processes only recent events). Queries combine results from both. Named after the Greek letter λ because the two paths branch from one input and meet at the output.
Lambda architecture — concrete example (not just typeahead)

Imagine a dashboard showing "YouTube views per video over the last 30 days".
  • Batch layer: every night at 2 AM, a Spark job reads the last 30 days of view-events from S3, groups by video-id, sums them, writes the result to a table views_30d. Takes 2 hours. Result is accurate but ~24 hours stale.
  • Speed layer: a Flink job consumes the same view-events live from Kafka and keeps a running counter in Redis: views_today[video-id]. Always within seconds of real-time but only covers "today".
  • Serving: dashboard query = views_30d[video] + views_today[video]. You get the best of both — yesterday's accurate snapshot plus today's live delta.
For our typeahead, the batch layer = the hourly trie rebuild; the speed layer = the Redis recent-counter that trie servers blend in every 30 s.
Slow batch (batch processing) = collect lots of records, then process them all together at scheduled intervals (every hour / every night). "Slow" not because each machine is slow, but because you wait for the next scheduled run before results appear. Trade-off: huge throughput and full accuracy, but freshness measured in hours. Opposite of stream processing, which handles each event the moment it arrives.
Spark = a distributed batch-processing framework (originally from Berkeley AMPLab, now Apache). You write code in Python/Scala/SQL like df.groupBy("query").count() and Spark splits it across hundreds of worker machines, each crunching a chunk of the input in parallel. Used for our hourly trie rebuild because it's great at "read a huge file, aggregate, write a result". Examples of users: Netflix, Uber, Airbnb.
Flink = a distributed stream-processing framework (Apache, originated at TU Berlin). Same idea as Spark but optimized for processing each event as it arrives (millisecond latency), with built-in windowing ("sum events in 5-minute tumbling windows") and exactly-once state. Used for our speed layer — the worker that updates the Redis recent-counter. Spark also has streaming, but Flink is generally considered stronger for true real-time.

So: Kafka is the buffer / source of truth for events, S3 is the long-term archive, the in-memory trie is the serving copy, and Redis is the speed-layer patch. Each storage is picked for what it's best at — Kafka for durable ordered ingest, S3 for cheap bulk reads, RAM for low-latency lookups, Redis for fast small writes.

Batch path (every 1 hour)
  1. Kafka topic queries.raw contains every completed search of the day (or longer — depends on retention).
  2. A Spark job reads the last 30 days of events from S3 (older data already off-Kafka).
  3. Aggregates: SELECT query, COUNT(*) FROM events WHERE ts > NOW() - 30d GROUP BY query.
  4. Applies frequency decay so older searches matter less.
  5. Builds a fresh trie offline; computes top-5 at every node bottom-up (a node's top-5 = merge of its children's top-5).
  6. Ships the new trie file to each trie shard, which hot-swaps it in. Old trie keeps serving until the new one is ready.
Frequency decay = giving more weight to recent searches and less to old ones. So that "covid" doesn't dominate suggestions forever. Common formula: weight = 0.5 ^ (age_in_days / half_life_days) with a 30-day half-life.
Hot-swap = replace a live data structure with a new version atomically while serving continues. Typical implementation: load the new trie into memory, build it fully, then flip a single atomic pointer from currentTrie → newTrie. Free the old one after in-flight requests drain. No downtime.
Stream path (real-time, continuous)
  1. A Flink job reads queries.raw live from Kafka.
  2. For each completed query (e.g., "coventry city"), writes to one Redis sorted set per prefix: ZINCRBY trending:c "coventry city" 1, ZINCRBY trending:co "coventry city" 1, … for all prefixes. Each set has a 2h TTL.
  3. Typeahead Service, when computing results for prefix "cov", queries both:
    • Trie shard → pre-computed top-5 (historical, 30-day)
    • ZREVRANGE trending:cov 0 4 → top-5 trending NOW (real-time)
  4. Merges both lists, deduplicates, and re-ranks using a weighted blend: 0.7 × trie_score + 0.3 × redis_score.

Result: a brand-new trending query can appear as a suggestion within seconds — even if it never existed in the trie. Existing queries get boosted if they spike.

Stream processing = handling events one-at-a-time as they arrive, with millisecond latency. Kafka Streams, Flink, Spark Structured Streaming are the typical tools.

07Read Path & Caching

Three layers of caching make this fast and cheap:

  1. Browser — the client app caches the last few prefixes' responses for ~10s.
  2. CDN / Edge — Cloudflare/Fastly cache the top ~1M most popular prefixes for 30–60s. 90%+ hit rate in practice — the top 0.1% of prefixes get 90% of traffic (Zipf distribution).
  3. Trie service — only cache misses make it here. In-memory trie lookup, <1 ms.
Zipf distribution = a power-law where a few items get most of the popularity. For searches: "weather", "facebook", "youtube" dominate; the long tail is huge but rarely queried. Makes caching very effective.
Why it's fast: 90% of requests never touch our servers (CDN). The remaining 10% hit a single memory lookup. Average end-to-end latency < 30 ms.

08Extras — Misspellings, Personalization, Trending

Misspellings ("hello wolrd" → "hello world")

Use a BK-tree (Burkhard-Keller tree) over the query corpus. For a prefix that doesn't match, search BK-tree for nearby strings within edit distance ≤ 2.

Edit distance = number of single-character edits (insert, delete, replace) to turn one string into another. "world" vs "wolrd" = 1 edit (swap).
BK-tree = a tree designed for edit-distance lookups. Lets you find all strings within distance K in O(log n) average. Used by spell-checkers.
Personalization

Blend global top-5 with the user's own recent searches:

  • Per-user history stored in a key-value store: user:123:history → list of past queries.
  • At request time, fetch user's history (1 ms) and re-rank: score = α · globalFreq + β · userFreq.
  • Tune α/β online with A/B tests.
A/B test (online experiment) = a controlled experiment where you split live users randomly into two (or more) groups and serve each group a different version of the system, then measure a business metric to decide which version wins. For our case: group A gets ranking weights α=0.7, β=0.3, group B gets α=0.5, β=0.5. We measure click-through rate on suggestions for each group over 1–2 weeks. If group B's CTR is statistically significantly higher (e.g., p < 0.05), we roll those weights out to 100% of traffic. Avoids guessing — the users tell us what works. Common tooling: in-house feature flags, or platforms like Optimizely, LaunchDarkly, Statsig.
Trending — Real-time via Redis per-prefix sorted sets
The problem Redis solves: The trie is rebuilt hourly. If "coventry city" starts trending at 8:15 AM and the next trie deploys at 9:40 AM, there's a 1.5-hour blind spot. Worse: a brand-new term (never searched before) won't appear in the trie at all — it has no pre-computed top-5 entry. Redis fills both gaps with real-time data.

  ...all prefixes...
  ZINCRBY trending:coventry city  "coventry city" 1

  EXPIRE trending:cov 7200   (reset TTL to 2 hours on each write)

This is write-amplified — one search = N writes (N = query length). But each write is O(log K) where K = members in that prefix's set. Trade-off: expensive writes buy instant prefix lookups at read time.

Redis sorted set (ZSET) = a Redis data structure that stores members ordered by a numeric score. Internally backed by a skip list + hash map, so insert and rank lookups are O(log n). Perfect for "top-K by score" queries.
ZINCRBY = "increment the score of a member in a sorted set by a given amount; create the member with that score if it doesn't exist." ZINCRBY trending:cov 1 "coventry city" means: in the sorted set named trending:cov, add 1 to the score of "coventry city". If "coventry city" didn't exist, it's created with score 1.
ZREVRANGE = "return members ranked from index X to Y in REVerse order (highest score first)." ZREVRANGE trending:cov 0 4 = return the top-5 members by score. The Z prefix marks all sorted-set commands. REV = highest-first. RANGE 0 4 = positions #1 through #5.
Write amplification = writing the same information to multiple locations to speed up reads. Here: one search for "coventry city" writes to 13 sorted sets (one per prefix character). Expensive on writes, but reads become a single O(log n) lookup per prefix. Same principle as a database maintaining multiple indexes.

10½Common Questions & Misconceptions

Q1. "Read path vs write path separation" — does that mean two network requests per query?

No. A user keystroke is one HTTP request — the read path. Submitting a finished search is a separate request the browser already sends to the search backend (independent of typeahead). The Search Service then asynchronously publishes that completed query to Kafka. So from the user's perspective: every keystroke = 1 read request; one final search submit = 1 write side-effect. The two "paths" describe how our backend processes the two kinds of events, not extra round-trips from the browser.

Q2. If we query Redis sorted sets on every read, won't that increase latency?

Yes — it adds one network hop (~0.5 ms). We mitigate it three ways:

  • Parallel calls: Trie shard lookup + Redis ZREVRANGE fire simultaneously. Total latency = max(trie, redis) ≈ 1 ms, not sum.
  • Redis is fast: ZREVRANGE on a sorted set with ~1000 members returning 5 items = O(log N + 5) ≈ microseconds.
  • CDN absorbs 90%: Only 10% of requests even reach the Typeahead Service (and Redis). The merged result gets cached at CDN for 30–60s.
  • Deny-list is applied at build time, not query time. The static blocklist is baked into the trie when the hourly Spark job runs — offensive words simply don't exist in the served trie.
  • Personalization fetch (user history) adds ~1 ms but runs in parallel with the trie+Redis lookups. If we exceed our 30 ms p99 budget, degrade gracefully (skip personalization).

Rule of thumb in design: parallelize, cache aggressively, and degrade gracefully.

Q3. Can we just store the trie in the CDN?

No, but we store the responses in the CDN. A CDN edge is a key→value cache for HTTP responses — it doesn't run trie-walk logic, it just returns whatever bytes we previously cached under a URL. So we cache the output of typeahead requests (e.g., GET /suggest?q=ca → top-5 JSON) at the edge for 30–60 s. The trie itself stays on origin servers because:

  • 500 GB doesn't fit on edge nodes, which are tuned for many small objects.
  • CDNs are dumb byte caches — no custom code, no in-memory data structures.
  • You'd need to push trie updates to hundreds of edge PoPs hourly = expensive and complex.
Q4. Can we put the trie in Redis instead of in-process RAM?

Technically yes — Redis can hold key-value pairs like prefix:"ca" → ["car","can",...]. But it's a worse choice here:

In-process RAM (current)Redis
Lookup latency< 0.1 ms (pointer chase, same process)0.5–1 ms (TCP round-trip per call)

When Redis IS a good fit: the speed layer (the recent-counter HASH), per-user history, trending sorted set. Small hot data, frequent small writes — exactly Redis's sweet spot. The big trie stays in-process.

Q5. What does "replicated for HA" actually mean?

HA = High Availability. "Replicated for HA" means each shard's data exists on multiple machines so that if one dies, another keeps serving with no data loss and minimal downtime. For our trie cluster:

  • Each of the 50 shards is copied onto 3 separate trie servers (typically in 3 different availability zones / racks).
  • One copy is the "primary"; the other two are "replicas" kept in sync. (For read-only data like the served trie, all 3 are equal — they all just hold the same trie file shipped by the builder.)
  • The load balancer health-checks each replica every few seconds. If one fails its health check, traffic auto-fails-over to a healthy replica within ~1 s.
  • Read scaling bonus: 3 replicas means 3× the read capacity of one shard for free.

Without replication, losing one machine means losing 1/50th of the trie until a new one is rebuilt and reloaded — minutes of partial outage. With 3 replicas, two more would need to die simultaneously for any user impact.

Q6. When does data actually get pushed into the cache?

There are two caching styles. We use the first:

  • Cache-aside (lazy / pull, what we use for the CDN edge): the cache is empty initially. A request comes in → cache miss → request goes to origin → response flows back through the CDN, which stores it on the way out with the TTL from the Cache-Control header. The very next request for the same URL is a hit. Self-warming: only prefixes anyone actually asks for end up cached. Downside: the first request is always slow.
  • Write-through / push (what we use for the trie): the Spark builder pushes the freshly built trie to all 150 trie pods at the end of each rebuild — they don't "fetch on demand". This is appropriate for the trie because it changes infrequently (hourly) and on a known schedule.

Optional cache warming: at deploy time we can pre-load the CDN with the top 10k most popular prefixes by sending synthetic requests through it. Avoids a cold-start latency spike right after a flush.

Cache TTL (Time To Live) = how long a cache entry stays valid before it's discarded or revalidated. Set via the Cache-Control: max-age=60 header from the origin response. After 60 s, the CDN treats the entry as expired and goes back to origin on the next request.

1230-Second Pitch

The Pitch

For ~10k QPS of typeahead, the read path is a trie with pre-computed top-5 at every node, sharded by first-2-letter prefix across ~50 shards with 3 replicas each, all in RAM. Each lookup is O(prefix length), <1 ms.

A CDN edge cache absorbs 90% of traffic — search prefixes follow a Zipf distribution, so a small set is overwhelmingly hot.

The write path is completely separate. Completed searches flow into Kafka, and a Spark/Flink batch job rebuilds the trie every hour with frequency decay. A stream layer patches in last-hour data via Redis counters for freshness.

Extras: BK-tree for misspellings, per-user history blending for personalization, deny-list for abuse. AP over CP — stale-by-minutes is fine; downtime isn't.

∞Glossary

A/B test

Show version A to half your users, B to the other half, measure which performs better. Used to tune ranking weights without guessing.

AP vs CP (CAP theorem)

AP = always responds even if data is slightly stale. CP = always correct but may refuse requests. Typeahead picks AP.

API Gateway

Single entry point for clients. Handles auth, rate-limit, routing.

BK-tree

A tree designed for finding strings within a certain edit distance. Used for spell-checking and misspellings.

Batch processing

Handle data in large groups on a schedule (e.g., once an hour). Accurate but slow.

CDN

Content Delivery Network — globally distributed cache servers close to users. Cloudflare, Fastly, Akamai.

Debounce

Wait for the user to pause typing before sending the network request. Reduces traffic.

Deny-list

Curated list of forbidden words/phrases never to suggest.

DFS

Depth-First Search — walk a tree by going deep before backtracking.

Edge cache

A cache at the CDN layer, close to users. Catches popular requests before they hit your backend.

Edit distance

Number of single-character changes to turn one string into another.

Frequency decay

Weight recent searches more than old ones, usually with exponential half-life.

Hot-swap

Replace a running data structure (like the trie) with a new version atomically, with no downtime.

Kafka

Distributed append-only queue. Used to move events between services durably.

Lambda architecture

Combining a slow accurate batch layer with a fast approximate stream layer.

p99 latency

The 99th percentile response time. p99 < 100 ms = 99% of requests under 100 ms.

QPS

Queries Per Second.

Redis sorted set

A Redis data structure keeping members ordered by a numeric score. Great for rankings and trending lists.

Replica

A copy of a shard for redundancy and read scaling.

Shard

One slice of a larger dataset living on a different machine.

Spark / Flink

Distributed data-processing frameworks for huge batch or stream jobs.

Stream processing

Handle events one at a time as they arrive. Fast but approximate.

Top-K

The K most popular/relevant items. Pre-computed so reads are instant.

Trie

Tree where each path from root spells a prefix. Each edge is one character.

TTL

Time To Live — auto-expiry timer on a cache key.

Typeahead

The dropdown of suggested completions as you type into a search box.

Zipf distribution

A power-law where the k-th most popular item gets frequency ≈ 1/k of the most popular. Concretely: the #2 query gets ~half the traffic of #1, the #10 gets ~1/10, the #100 gets ~1/100. So the top 0.1% of prefixes can easily generate 90% of all requests — which is exactly why caching just a few thousand entries deflects almost all our traffic.
