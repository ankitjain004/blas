01Problem & Requirements

Goal. Let creators upload videos and let users watch smoothly on any device/network, with low startup delay and high availability.
  • Upload huge videos reliably (resumable, multi-part).
  • Transcode into multiple resolutions/bitrates.
  • Serve with adaptive bitrate (Auto quality mode).
  • Search, recommendations, likes, comments, subscriptions.
  • Moderation, copyright checks, abuse controls.

02Capacity Math

Notation. B=byte, b=bit, 1 B = 8 b.
  1. Uploads: 10M videos/day × 500 MB avg = 5,000,000,000 MB/day = 5 PB/day raw ingest.
  2. With multi-resolution transcoding (avg 1.6x storage overhead) ≈ 8 PB/day net written.
  3. Viewers: 10M concurrent × 2 Mbps average stream = 20 Tbps egress.
  4. If CDN hit ratio is 95%, origin egress drops to about 1 Tbps.
  5. Segment size around 2-4 s chunks; billions of segment GETs/day.

04Upload + Processing Pipeline

  1. Client requests upload session; receives pre-signed multipart URLs.
  2. Client uploads chunks directly to blob storage (bypasses app servers).
  3. Upload complete event enqueued for processing.
  4. Transcoder creates renditions: 144p, 240p, 360p, 480p, 720p, 1080p, optional 4K.
  5. Generate HLS/DASH manifests and thumbnails.
  6. Run moderation and copyright checks before publishing.
ABR renditions are multiple encodings of the same video with different bitrate/quality levels so the player can switch dynamically.

05Playback Path & ABR

  • Player first fetches manifest, then downloads short segments (2-4 s).
  • Auto quality estimates throughput from segment download rate and buffer health.
  • If buffer high + throughput stable, step up bitrate; otherwise step down quickly.
  • CDN serves most segment traffic; origin serves misses and long-tail content.

06Metadata, Search, Recommendation

  • Metadata DB stores title, channel, tags, language, visibility.
  • Search index (Elasticsearch/OpenSearch) handles text queries and filters.
  • Recommendation service uses watch history, embeddings, popularity, freshness.
  • Counter service for views/likes with stream aggregation and anti-fraud filters.
Fanout-on-read for home feed is common at YouTube scale because precomputing for every user can be expensive with volatile trends.

07Reliability and Cost Controls

  • Multi-region object storage replication for disaster recovery.
  • Queue-based backpressure for transcoding bursts.
  • Tiered storage: hot recent videos on faster tiers; cold on cheaper archival tiers.
  • CDN cache key tuning and pre-warm for expected viral uploads.
  • Graceful degradation: fallback to fewer bitrates if transcoder lag rises.

0930-Second Pitch

YouTube is a CDN-first video platform. Creators upload directly to object storage via resumable multipart sessions. A queue-driven transcode pipeline creates ABR renditions and manifests for HLS/DASH playback. Viewers fetch manifests and short segments from edge caches, while ABR logic in the client adapts quality based on throughput and buffer to minimize stalls. Metadata/search/recommendation run as separate services over event streams. Reliability comes from multi-region replication, queue backpressure, and graceful degradation under spikes.

10Glossary

ABR

Adaptive Bitrate Streaming; player switches bitrate based on measured network + buffer.

HLS/DASH

Streaming formats with manifests pointing to small media segments.

Manifest

Playlist listing available renditions and segment URLs.

Origin

Primary backend storage/service behind the CDN on cache miss.

Transcoding

Re-encoding uploaded video into multiple codecs/resolutions/bitrates.

Rebuffering

Playback stall when buffer runs dry before next segment arrives.
