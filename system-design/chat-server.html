01Problem & Requirements

The ask. Design a chat system like Facebook Messenger — 1:1 and group messaging, low latency delivery, online presence indicators, multi-device support, and push notifications for offline users.
Functional
  • 1:1 chat with low delivery latency (< 500 ms for online recipients).
  • Group chat with a maximum of 100 members.
  • Online presence — see who is online / last seen.
  • Multi-device — same account logged in on phone + laptop + tablet simultaneously.
  • Push notifications — notify offline users.
  • Persistent history — messages stored forever.
  • Max message size: 100,000 characters.
Non-Functional
  • 50M DAU.
  • Latency: delivery to online recipient p99 < 500 ms.
  • Ordering: messages within a conversation appear in send order.
  • Durability: never lose a delivered message.
  • Availability > Consistency: it's better to deliver a message twice (dedup on client) than drop it.
Out of Scope (mention as extensions)
  • End-to-end encryption (E2EE) — server can't read content, loses search/moderation capability.
  • Media (images, video, voice) — separate upload path + CDN.
  • Read receipts / typing indicators — ephemeral signals, similar to presence.

02Capacity Math

Notation. 1 char = 1 B (UTF-8 ASCII). 1 KB = 10³ B.
  1. Concurrent connections = 50M DAU × 10% active × 1.5 devices = ~7.5M WebSocket connections.
  2. Messages/day = 50M × 40 msg/user = 2B messages/day.
  3. Messages/sec (avg) = 2B / 86,400 ≈ 23K msg/s. Peak (3×) = ~70K msg/s.
  4. Average message size ≈ 200 B (most messages are short; 100K is the cap).
  5. Storage/day = 2B × 200 B = ~400 GB/day.
  6. Storage/year = 400 GB × 365 = ~146 TB/year (before replication).
  7. Fan-out (groups): avg group size ~10, peak group msg rate ~7K msg/s × 10 = ~70K deliveries/s from groups alone.
  8. Total delivery throughput: ~23K (1:1) + ~70K (group fan-out) = ~100K deliveries/s peak.

03Choosing the Protocol

OptionHowVerdict
HTTP Polling Client asks "new messages?" every N seconds. ❌ Wasteful at scale (7.5M clients × every 3s = 2.5M QPS of mostly-empty responses). Latency = N/2 avg.
HTTP Long Polling Client holds connection open until server has data or timeout. ⚠️ Better but still one-directional. Hard to push server-initiated events (presence, typing).
WebSocket Persistent full-duplex TCP connection. Server pushes instantly. ✅ Best fit. Bidirectional, low latency (~RTT), efficient for presence + messages. 10 KB RAM per connection.
Pick WebSocket for the real-time message path. Use plain HTTP only for control-plane operations (login, fetch history, create group, upload profile).
WebSocket = a protocol (RFC 6455) that upgrades an HTTP connection to full-duplex, persistent TCP. One connection handles both sending and receiving — no polling overhead.

Two paths: the real-time path (WebSocket → LB → WS Gateway → Chat Service) carries live messages; the control path (HTTP REST → API Gateway → User/Group/History services) handles login, group management, and history fetches.
WebSocket Gateway = stateful tier that holds persistent TCP connections. Each pod handles ~50K connections. On connect, registers (user_id, connection_id, pod) in Redis so other services know where to push.
API Gateway (L7) = thin reverse-proxy tier (Nginx / Envoy / Kong) for the control path. It does cross-cutting concerns only — routing (path → backend), authentication (validate JWT), rate limiting, TLS termination. No business logic runs inside it; it forwards to User/Group/History services. Real-time WebSocket traffic bypasses it and goes straight to the WS Gateway.
Service Discovery = registry (Consul / etcd / Kubernetes DNS / Eureka) that maps a logical service name to live pod IPs. Gateways and services query it (or use client-side LB) to find healthy instances as pods scale up/down — "how does the gateway find the Group Service?"
Chat Service = stateless tier that validates, persists, and enqueues messages. Horizontally scalable — any instance can handle any message. Does authorization (business rules), while the gateway did authentication.
Delivery Worker = Kafka consumer that performs fan-out: looks up recipient sessions, decides push path (WebSocket or notification), and delivers. Decouples send-ack from fan-out latency.

05Message Flow — 1:1 Chat

  1. Alice types "Hey Bob" and sends over her WebSocket.
  2. Gateway receives the frame, routes to Chat Service.
  3. Chat Service:
    • Validates (auth, rate-limit, msg size ≤ 100K).
    • Generates msg_id via Snowflake (time-sortable 64-bit ID).
    • Writes to Cassandra messages table (partition = channel_id).
    • Publishes to Kafka topic msg.delivery.{hash(bob_id)}.
    • Acks back to Alice via Gateway: ✓ (sent, msg_id).
  4. Delivery Worker consumes the Kafka event:
    • Looks up Bob's sessions in Redis: user:bob:sessions.
    • Bob ONLINE (2 devices) → pushes message to both Gateway pods holding Bob's connections.
    • Bob OFFLINE → enqueues to Push Notification Service.
    • Increments unread_count for Bob in Redis.
  5. Bob's devices receive the message via WebSocket → render immediately.
End-to-end latency (Alice → Bob): typically 100–300 ms. The ack to Alice is independent — she doesn't wait for Bob to receive.

06Message Flow — Group Chat

  1. Alice sends to group "Engineering" (45 members).
  2. Chat Service:
    • Validates Alice is a member of the group.
    • Generates msg_id, persists once (partition = group_channel_id).
    • Looks up group membership: 45 user_ids.
    • Publishes 44 delivery events to Kafka (one per recipient, excludes sender).
    • Acks to Alice.
  3. Delivery Workers (parallel consumers): for each of 44 recipients, same logic as 1:1 — lookup sessions, push if online, notify if offline.
Why fan-out on write (not read)? Groups are small (≤100). Writing 100 delivery events is cheap. The alternative — "fan-out on read" — would mean every user polls and merges messages from all their groups, which is costlier at read time for a real-time system.
Message stored once, delivered many. The message body lives in one Cassandra row (keyed by group_channel_id). Delivery events in Kafka are lightweight pointers (msg_id + recipient_id). We do NOT duplicate the message body per recipient.

07Multi-Device Sync

Alice has 3 devices (phone, laptop, tablet). All connected via separate WebSockets to potentially different Gateway pods.

Session Registry (Redis)
user:alice:sessions = {
  "conn_abc": { device: "iphone",  gateway_pod: "gw-17", connected_at: 1717000000 },
  "conn_def": { device: "laptop",  gateway_pod: "gw-42", connected_at: 1717000100 },
  "conn_ghi": { device: "tablet",  gateway_pod: "gw-08", connected_at: 1717000200 }
}
Incoming message delivery

When a message arrives for Alice, the Delivery Worker pushes to all 3 connections. Each device renders it.

Read sync (cross-device)
  1. Alice reads msg_42 on her phone.
  2. Phone sends {type:"read", channel_id, msg_id:42} to Chat Service.
  3. Chat Service updates read_cursor(alice, channel_id) = 42.
  4. Chat Service pushes sync event to Alice's other devices (laptop, tablet).
  5. Those devices update UI: mark messages up to 42 as "read".
Reconnection after offline
  • Each device tracks its last_received_msg_id per channel.
  • On reconnect, device sends sync request: "give me messages after X for each of my channels."
  • Chat Service fetches from Cassandra: SELECT * FROM messages WHERE channel_id = ? AND msg_id > ? LIMIT 50.
  • Paginates through any backlog.

Heartbeat = periodic "I'm alive" signal from client to server. 30s interval balances between accuracy and network cost. Missing 2 heartbeats (60s) = offline.
User group presence fan-out. In a group of 100, showing all 100 members' presence means 100 subscriptions per viewer. Optimization: only show presence for members who are in the visible scroll area, batch presence lookups.

09Push Notifications

  1. Trigger: Delivery Worker finds recipient has NO active WebSocket sessions in Redis.
  2. Push Notification Service:
    • Looks up user's registered device tokens (APNs for iOS, FCM for Android).
    • Builds payload: { title: "Alice", body: "Hey Bob", badge: unread_count }.
    • Sends to APNs/FCM.
  3. Rate-limiting / collapsing: If 5 messages arrive in 10 seconds, collapse into one notification: "Alice sent 5 messages."
Race condition: User comes online between message-persist and push-send. Mitigation: check presence one final time before sending push. Accept rare duplicates (client deduplicates by msg_id).
Device Token Management
  • Stored in MySQL: push_tokens(user_id, device_id, platform, token, updated_at).
  • Tokens refresh when app opens — update on every login.
  • Invalid tokens (APNs rejects) → mark disabled, stop sending.

Why Cassandra for messages?
  • Write-heavy: 23K writes/s → Cassandra excels at sequential writes (LSM tree).
  • Partition by channel_id: all messages in one conversation on one node → fast range scans for "load history".
  • No cross-partition queries: you always access messages within one channel.
  • Time-sorted by msg_id (Snowflake): natural chronological order without extra index.
Message ID: Snowflake
64-bit Snowflake ID:
┌────────────────────┬──────────┬────────────┐
│ 41 bits timestamp  │ 10 bits  │ 12 bits    │
│ (ms since epoch)   │ machine  │ sequence   │
│ ~69 years range    │ 1024     │ 4096/ms    │
└────────────────────┴──────────┴────────────┘

Properties:
- Time-sortable (older messages have smaller IDs)
- Globally unique (no collisions across machines)
- 4M IDs/sec per machine → far exceeds 70K msg/s
channel_id = unique identifier for a conversation. For 1:1: deterministic hash of sorted(user_a, user_b). For groups: the group_id itself.

13Interview Talking Points

  1. WebSocket is the only sane choice for real-time bidirectional messaging at this scale.
  2. Gateway is stateful, Chat Service is stateless — separation lets you scale independently.
  3. Kafka decouples send from delivery — sender gets instant ack; fan-out happens async. Critical for group messages.
  4. Cassandra partitioned by channel_id — write-heavy workload, sequential reads for chat history.
  5. Snowflake IDs — time-sortable, globally unique, no coordination needed.
  6. Multi-device = multiple session entries in Redis per user. Deliver to all; sync read state across.
  7. Presence uses heartbeat + TTL — not stored in DB. Redis key expiry = offline.
  8. Push notifications only when offline — check presence before sending to avoid duplicates.
  9. Fan-out on write for groups — groups are small (≤100), so write amplification is bounded.
  10. Trade-offs: at-least-once delivery (client dedup), eventual consistency on presence, storage growth for persistent history.

1430-Second Pitch

A chat system at 50M DAU is fundamentally WebSocket connections for low-latency push + Kafka for async fan-out + Cassandra for durable message storage. Clients maintain persistent WebSocket connections to a Gateway layer; the Gateway registers sessions in Redis so the Chat Service knows where to route messages. 1:1 messages go directly; group messages fan out through Kafka delivery workers to all members' active sessions. Multi-device works by registering multiple connections per user and syncing read state across them. Presence uses Redis key TTL + heartbeats — no heartbeat for 60s = offline. Push notifications fire only when no active WebSocket exists. Everything is partitioned by channel_id so one conversation's messages live on one Cassandra partition for fast sequential reads.

15Glossary

WebSocket

Full-duplex TCP protocol (RFC 6455) for persistent bidirectional communication — server pushes instantly without polling.

Snowflake ID

64-bit time-sortable unique ID: 41 bits timestamp + 10 bits machine + 12 bits sequence. ~4M IDs/sec/machine.

Fan-out on Write

When a message is sent, immediately create delivery events for each recipient. Good for small groups; expensive for broadcast (1M followers → use fan-out on read).

channel_id

Unique identifier for a conversation. For 1:1: hash(sorted(userA, userB)). For groups: the group_id.

Session Registry

Redis hash mapping user_id → list of active WebSocket connections (device, gateway_pod, connected_at). Enables multi-device delivery.

Heartbeat / TTL Presence

Client pings every 30s; Redis key TTL is 60s. No ping for 60s → key expires → offline. No DB writes needed.

Sync Cursor

Per-device last_received_msg_id. On reconnect, fetch all messages after cursor to catch up.

Idempotent Delivery

Client deduplicates by msg_id. At-least-once from Kafka is safe because rendering the same message twice is prevented client-side.

APNs / FCM

Apple Push Notification Service / Firebase Cloud Messaging — platform-specific push notification gateways for iOS/Android.

E2EE (Extension)

End-to-end encryption where server relays only ciphertext. Loses server-side search and content moderation.

Long Polling (alternative)

Client holds HTTP connection open until data arrives. Simpler than WebSocket but unidirectional and less efficient for frequent updates.

Cassandra Partition

All rows with the same partition key live on the same node. Messages partitioned by channel_id → fast sequential reads for chat history.
