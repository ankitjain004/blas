Centralized Logging at Realistic Scale

00Problem & Scale Check

The setup (realistic mid-to-large SaaS company)
  • ~5,000 backend service instances (pods/VMs running across Kubernetes clusters in 3 regions). Comparable to Stripe / Wise / Shopify in 2024–26.
  • Each instance emits ~200 log messages every second on average — mostly small structured JSON lines like {ts, level, service, traceId, msg, ...}.
  • Average message size ≈ 500 B after JSON serialization, before compression.
  • Goal: collect all of them and let engineers/tools read them back in order for debugging, audits, dashboards, alerting.
Do the math first
5,000 instances  ×  200 messages/sec  =  1,000,000 messages/sec   (1 M/sec)

Message size                  ≈ 500 B  (uncompressed JSON, 1 char = 1 B in ASCII)
Ingestion bandwidth (raw)      = 10⁶ × 500 B          = 5 × 10⁸ B/s = 500 MB/s
With ~5× compression on wire   = 500 MB/s / 5         = 100 MB/s on wire

Per day (uncompressed at rest):
  500 MB/s × 86,400 s          = 4.3 × 10¹³ B ≈ 43 TB/day
  After columnar compression
  in cold tier (≈ 8×)          ≈ 5–6 TB/day on S3

Peak factor (incident spikes,  ~3–5× base)            → size to ~3M msg/sec headroom
Why this scale matters for the interview. 1M msg/s is the sweet spot: too big for a single Kafka broker, a single ES cluster, or a single Postgres table — so you have to partition, tier, and bulkhead. But it's not Google-scale fantasy — every senior engineer at a real product company will hit numbers like this once their product grows. The same architecture also scales up by 100× just by adding partitions and brokers, which is why interviewers like the problem.
What is a "log message"? A short text record a service writes whenever something happens — for example 2026-05-25 10:14:03 INFO svc=auth user=42 traceId=abc action=login latency_ms=87. Engineers read these later to figure out what went wrong (debugging) or to answer "what did the system do?" for an audit.
Notation we'll use = b = bit (lowercase), B = byte (uppercase); 1 B = 8 b. 1 KB = 10³ B, 1 MB = 10⁶ B, 1 GB = 10⁹ B, 1 TB = 10¹² B (decimal/SI). Storage and ingest are always in bytes. Network NIC speeds (1 Gbps, 10 Gbps) are in bits — divide by 8 before comparing.

01Clarify "in order"

Before designing anything, pin down what "in order" means. The cost differs by 1000×.

Ordering scopeWhat it meansCostRealistic?
Global total orderEvery message worldwide gets a single rank: 1st, 2nd, 3rd...Astronomical — every server would have to agree on every message before storing it❌ No
Per-server FIFOMessages from server S are read in the order S sent them. Server A and Server B aren't compared.Cheap — just partition by server-id✅ Strong default
Causal / per-trace orderIf event A caused event B (across servers), A is shown before BMedium — needs logical clocks✅ When tracing requests
Approximate time orderWithin a few seconds of each otherEasy — sort by timestamp with a tolerance✅ Common compromise
FIFO = "First In, First Out". Messages come out in the same order they went in — like a queue at a coffee shop.
Partition = a split / chunk. If 100M messages/sec land in ONE big bucket, the bucket overflows. If we split them into 5,000 smaller buckets, each bucket only sees ~20k/sec — manageable. "Partition" is just a fancy word for "one of those smaller buckets."
Recommended answer: Guarantee per-server FIFO (cheap, free with Kafka partitioning) + attach a Hybrid Logical Clock on each message for causal cross-server reading when needed.

Ingest tier = the "front door". A pool of small machines that receive incoming messages. Their only job is: authenticate the sender, batch a few messages together, and forward them to the next layer. They keep zero state — easy to add/remove.
Kafka = a giant, durable, append-only logbook. You write messages to the end; consumers read them in the same order. It's split into "partitions" (those buckets again) so it can handle huge volume. Used by LinkedIn, Uber, Netflix.
Broker = one machine in the Kafka cluster. ~200 brokers means 200 physical/virtual servers that together form the Kafka system. Each broker holds a slice of the data.
Consumer = anyone reading from Kafka. For us, consumers are the components that pull messages out and put them into the storage tiers (hot/search/cold).
NVMe SSD = the fastest type of disk in 2026. Think of it as ~10× faster than a regular SSD. Used for the "hot" tier where humans tail logs in real time.
S3 Parquet = cheap, durable object storage (S3 = Amazon's storage service) holding data in Parquet format. Parquet is a compressed columnar file — like a super-efficient CSV. Used for long-term archive and analytics.

03Component Sizing

3a · Sidecar agent (the source)

Every service instance runs a small sidecar called an agent (Fluent Bit, Vector, or in-house). Its job:

  • Batch ~100 messages together (every 10–100 ms) instead of sending one-by-one — saves network calls.
  • Stamp each message with: serviceId, podId, a local sequence number that goes up by 1 every msg, wall clock, and an HLC (Hybrid Logical Clock).
  • Write to local disk first in a small ring buffer (≈ 100 MB) — if the network dies for 30 seconds, nothing is lost.
  • Compress using Zstd — shrinks JSON ~5×.
  • Send to the nearest ingest pod ("sticky" — always the same one, for ordering).

Per-instance outbound bandwidth: 200 msg × 500 B / 5 (compression) = 20 KB/sec. Trivial.

Agent = a small daemon (a background program) running on each server. It's the courier that hands off messages from the app to the central system.
Ring buffer = a fixed-size circular file or memory area. When it fills up, the oldest entries get overwritten by new ones. Useful as a tiny safety net during network outages.
Compression = squeezing the data smaller. 500-byte messages often have a lot of repeated structure (timestamps, levels, similar fields). Snappy and Zstd are two popular fast compressors — they shrink data so we transmit less.
3b · Edge ingest tier

Each ingest pod can sustain ~25,000 msg/sec → we need ~50 ingest pods (with 50–70% headroom for incident spikes). They:

  • Authenticate the sending agent via mTLS so random clients can't inject fake logs.
  • Route to a stable Kafka partition using hash(serviceId + podId) % numPartitions — same source always lands on same partition → preserves order.
  • Apply back-pressure: if Kafka is slow, hold the message briefly rather than dropping it.
Hash = a function that turns any input into a fixed-size number. hash("auth-svc:pod-42") = 8137264291. Used here to evenly spread sources across partitions deterministically.
Back-pressure = a signal that says "slow down, I'm full". When the downstream (Kafka) is overloaded, the upstream (ingest) slows down so nothing crashes.
mTLS (mutual TLS) = a TLS handshake where both sides present certificates, not just the server. Standard way services authenticate each other inside a private network without sharing API keys.
3c · Kafka cluster — the ordered logbook

This is the heart of the design. Sizing math:

  • Each Kafka partition sustains ~50–100 MB/sec of writes.
  • Total write volume (uncompressed): 500 MB/sec ÷ 50 MB/sec ≈ 10 partitions minimum. We pick 500 partitions for parallelism, future growth, and one-partition-per-consumer-thread sizing.
  • RF=3 for durability → disk I/O is 1.5 GB/sec aggregate.
  • A typical Kafka broker handles ~100–150 MB/sec sustained → ~20 brokers with headroom.
  • Partition key = serviceId + podId — all messages from one pod land on the same partition, preserving per-pod FIFO.
  • Retention: 72 hours hot in Kafka. After that, consumers should have shipped data to cheaper tiers.
Replication factor (RF) 3 = each message is stored on 3 different brokers. If one broker dies, two copies still exist. Cost: 3× disk usage, but you can lose 2 machines before losing data.
Kafka Mental Model — Broker, Partition, Consumer, RF

These four concepts connect tightly. Here's how they fit together:

Broker = one physical/virtual server in the Kafka cluster. It has CPU, RAM, and disks. It stores a subset of all partitions. Each broker handles ~100–150 MB/s of sustained writes. Our cluster has ~20 brokers, which together handle the full 500 MB/s (× RF=3 = 1.5 GB/s total I/O).
Partition = one ordered, append-only log file. Messages go in at the end; consumers read from beginning to end, always in order. The partition is the unit of parallelism — more partitions = more consumers can work in parallel. Each partition has ONE leader broker (handles reads/writes) and RF−1 replica brokers (passive copies). We have 500 partitions → 500 consumers can read in parallel.
Consumer = any program that reads messages from Kafka partitions. It "subscribes" to one or more partitions and reads them sequentially. In our system, the consumers are the background processes that write data into the Hot store, Search index, and Cold archive. Each consumer processes messages in the exact order they were written — that's how we preserve per-server FIFO.
RF (Replication Factor) = how many total copies of each partition exist across different brokers. RF=3 means 1 leader + 2 replicas = 3 copies on 3 separate machines. If the leader dies, one replica is promoted to leader in <1 second (ISR election). You can lose 2 brokers and still have the data. Trade-off: 3× disk usage and 3× network replication traffic. For a logging system where data loss = lost debugging ability, RF=3 is the standard choice.

Tiered storage = expensive fast disks for recent data, cheap slow disks for old data. Like keeping yesterday's mail on your desk and a year-old tax form in the attic.
NVMe node = a server equipped with NVMe SSDs (the fastest disk type, ~7 GB/s read, ~1M random IOPS via PCIe). 04How Ordering Actually Works

Per-server FIFO — free, from Kafka

Each message carries:

{
  "serverId":   "srv-42",
  "localSeq":   18394201,        // server's own counter; goes up by 1 each msg
  "wallClock":  1716636843123,   // server's local time in milliseconds
  "hlc":        [1716636843123, 7],  // Hybrid Logical Clock
  "payload":    "user=42 action=login ..."
}

Because we partitioned by serverId, all messages from srv-42 are stored in one Kafka partition in the order they arrived. Kafka guarantees a consumer reads them in the same order.

Detect drops or duplicates

The consumer remembers lastSeenSeq[serverId]. If the next message has localSeq ≠ lastSeen + 1, we know we missed something → alert.

Hybrid Logical Clock (HLC) — the clever bit

Different servers have slightly different clocks (a few milliseconds off). If two events happen on different servers, whose timestamp do we trust? An HLC is a pair of numbers that combines wall-clock time with a logical counter, so causality is preserved even when clocks drift.

Hybrid Logical Clock (HLC) = (physicalTime, logicalCounter). The physical part is just the wall clock. The logical counter ticks up whenever two events happen in the same millisecond OR when a message arrives from a "future" server. Result: if event A really did happen before event B (because A's message was used to make B), then HLC(A) < HLC(B). Always.
When server emits an event:
   hlc.physical = max(local clock, last hlc.physical)
   hlc.logical  = (hlc.physical advanced?) ? 0 : hlc.logical + 1

When a server receives a message:
   hlc.physical = max(local clock, last hlc.physical, incoming hlc.physical)
   hlc.logical  = (something else, but you get the idea)

At read time, the reader pulls from N partitions and merges by HLC, with a small bounded delay (e.g., 5 seconds) to absorb skew. K-way merge is O(log N) per message — very fast.

Clock skew = the small difference between any two computers' clocks. Even with NTP (the standard time-sync protocol), two servers can be off by 1–50 milliseconds. HLC tolerates this gracefully.
K-way merge = picking the smallest item from K sorted lists at each step. Like merging K already-sorted decks of cards into one sorted deck. Uses a min-heap, takes O(log K) per pick.

05Handling Failures

FailureWhat happensMitigation

Availability Zone (AZ) = one datacenter (or set of datacenters with shared power/network) within a cloud region. "Cross-AZ replication" means data copies live in different datacenters — so a single fire/power-outage doesn't lose data.
ISR (In-Sync Replica) = a Kafka broker that has all the latest data and is ready to take over if the leader dies. Kafka leader election among ISRs happens in under a second.
Leader election = the process of picking which broker is "in charge" of a partition. If the current leader dies, the remaining ISRs vote and one becomes the new leader. Clients automatically reconnect.

06Reading the Data Back

Live tail (most common)

"Show me logs from srv-42 right now." → Consumer reads that one partition end-to-end, latency p99 < 1 second.

Search (by text or tag)

Background pipeline writes from Kafka → Elasticsearch/ClickHouse. Queries run there.

Audit / compliance

Pull months-old data from S3 Parquet files, partitioned by (serverId-prefix, hour).

Trace reconstruction (cross-server)

Given a traceId (an ID stamped on every log message in one user request), fan out queries to every partition that touched that trace, then K-way merge by HLC.

p99 latency = "99th percentile" latency. If you do 1000 reads, p99 is how fast the 990th-slowest of them was. p99 < 1s means almost everyone gets a fast response; only 1% wait longer.
Trace ID = a unique identifier (like abc-123-xyz) attached to every log message produced while handling a single user request. Even if 50 servers touched the request, you can grep all their logs by the trace ID to reconstruct the full story.

07Where Redis Fits (and doesn't)

Redis is NOT the primary store. 1M msg/sec global counter is impossible on a single Redis (single-thread bottleneck → max ~150k INCR/sec on one instance). But Redis has narrow roles:

RolePatternHow to scale
Per-server sequence counter (if agents are stateless)INCR seq:{serverId}Better: keep counter inside the agent. No Redis needed.
Rate-limiting noisy sourcesINCR rl:{serverId} + TTLShard by hash(serverId) across many Redis instances
Live dashboard countersSharded counter patternMany small counters → aggregate on read
Redis INCR = an atomic "add 1" command. Atomic = even if 1000 clients ask at once, every increment is counted exactly once. Each Redis instance handles ~100,000–200,000 INCRs per second.
TTL (Time To Live) = how long a key lives before Redis auto-deletes it. For rate limiting, you set TTL=1 second so the counter resets every second.
Sharded counter = splitting one logical counter into N physical ones to break the hot-key bottleneck. Each client picks a random shard 0..N-1, increments only that. To read the total, sum all N counters. Trade-off: read isn't perfectly atomic, but write throughput scales linearly.
Rule of thumb: One Redis instance ≈ 100k–150k INCRs/sec. For a 1M ops/sec global counter you'd need ~10 shards (and you're usually better off without a global counter at all).

09Interview Talking Points

  1. Always clarify "in order" first. Don't assume global total order — it's impossible at this scale.
  2. Partition by serverId — gives free per-server FIFO via Kafka.
  3. Hybrid Logical Clocks for causal cross-server ordering without global consensus.
  4. Batch at the source (10–100 ms windows) — 10× reduction in connection overhead.
  5. Tiered storage — hot/warm/cold is mandatory at PB scale.
  6. Numbers matter — show you sized brokers, ingest nodes, and disk.
  7. Redis is not the primary answer here — but it has narrow uses (rate limiting per source).
  8. Failure modes — agent disk buffer, Kafka RF=3, cross-AZ replication.

1030-Second Pitch

The Pitch

For a SaaS company at ~1M log msg/sec (5,000 service pods × ~200 msg/s), global ordering is unnecessary and unaffordable. I'd guarantee per-pod FIFO and use HLCs for cross-service causality.

Each pod runs a sidecar agent (Fluent Bit / Vector) that batches, persists to a local ring buffer, compresses with Zstd, and ships to a sticky edge-ingest tier (~50 pods). Edge routes by hash(serviceId+podId) to a Kafka cluster with ~500 partitions on ~20 brokers (RF=3, min.ISR=2) — Kafka gives per-partition order natively.

Consumers fan out to a hot NVMe store (72 h, ~50 TB), ClickHouse for text search and dashboards (7–30 days), and S3 Parquet for long-term archive. For causal queries across pods, readers k-way merge partitions by HLC with a small bounded delay.

Critical detail: a single global counter is impossible — but per-pod sequence counters live inside the agent, costing zero coordination. Same architecture scales up 100× just by adding partitions and brokers.

∞Glossary — All Terms in One Place

Every technical word from this doc, in plain English.

Agent

A small background program running on each server. Collects local log messages and ships them to the central system.

Atomic operation

An operation that either fully completes or doesn't happen at all — no half-states. INCR in Redis is atomic.

Availability Zone (AZ)

One datacenter inside a cloud region. AZs share a region but have independent power/network so an outage in one doesn't kill another.

Back-pressure

A "slow down, I'm full" signal sent upstream. Prevents fast producers from crushing slow consumers.

Batch

Group many small messages together before sending — saves network round-trips. Like carpooling.

Broker

One machine in a Kafka cluster. Holds a slice of the partitions.

Clock skew

Two computers' clocks disagreeing by a few milliseconds. Inevitable in distributed systems.

Compression

Shrinking data so it takes less space/bandwidth. Snappy and Zstd are popular fast compressors.

Consumer

Code that reads messages from Kafka (the opposite of a producer, which writes).

Edge ingest tier

The "front door" of the system. A pool of small machines that accept incoming messages and forward them on.

Elasticsearch

An open-source search engine for text data. Used for log search.

FIFO

"First In, First Out" — items come out in the order they went in. Like a queue.

Hash

A function that turns any input into a fixed-size number. Used to spread data across partitions deterministically.

HLC (Hybrid Logical Clock)

A pair (wallTime, counter) that preserves cause-and-effect order across servers even when clocks drift.

Hot / Warm / Cold tier

Storage layers ranked by speed and cost. Hot = fastest + most expensive (recent data). Cold = slowest + cheapest (old archives).

Ingest

To accept incoming data into a system.

ISR (In-Sync Replica)

A Kafka broker that has all the latest data and can take over if the leader dies.

Kafka

An open-source append-only logbook built for huge volume. Partitioned for parallelism. Used by most large companies.

K-way merge

Merging K sorted lists into one sorted list using a min-heap. O(log K) per pick.

Leader election

The automatic process of choosing which broker "owns" a partition when the previous owner dies.

Log message

A short text record a server writes when something happens (login, error, click, etc.).

NVMe SSD

The fastest type of disk in 2026 — roughly 10× quicker than a normal SSD.

p99 latency

The 99th percentile response time. p99 < 1s means 99% of requests are faster than 1 second.

Parquet

A compressed columnar file format — like a super-efficient CSV. Great for analytics.

Partition

One slice/bucket of a larger logical stream. Splits work across machines so no single node is overloaded.

Pipelining

Sending many commands without waiting for replies in between. Amortizes network cost.

Redis

An in-memory key-value store. Very fast (~100k–1M ops/sec). Used for caching, rate limiting, counters.

Replication factor (RF)

How many copies of each message Kafka keeps. RF=3 means 3 copies on 3 different brokers.

Ring buffer

A fixed-size circular buffer. New entries overwrite the oldest. Used as a small disk safety-net.

S3

Amazon's object storage service. Practically infinite, cheap, very durable. Used for long-term archive.

Sharded counter

One logical counter split into N physical counters to break the hot-key bottleneck.

Sharding

Splitting data/work across multiple machines so no one is overloaded.

Sticky routing

Always sending the same input to the same destination (e.g., same server-id always to same ingest node) to preserve order.

Throughput

How many operations per second the system handles.

Trace ID

A unique ID attached to every log line during one user request. Lets you reconstruct what happened across many servers.

TTL (Time To Live)

How long a key lives before being auto-deleted. Useful for ephemeral counters and caches.

Wall clock

The actual real-world time on a computer (vs. a logical counter). Subject to drift and NTP corrections.
