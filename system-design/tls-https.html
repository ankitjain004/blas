01What Is HTTPS & TLS?

TLS (Transport Layer Security) = the protocol that encrypts and authenticates data as it travels over a network, so nobody in between can read or tamper with it. It is the "S" in HTTPS. HTTPS = HTTP + TLS. The browser padlock is TLS at work.
HTTPS = HTTP running on top of a TLS-encrypted connection. Same HTTP requests/responses, but wrapped in a private, verified tunnel. SSL is the old, deprecated name for the same idea. People say "SSL certificate" out of habit, but modern systems use TLS (current standard: TLS 1.3, 2018).

THE PROBLEM IT SOLVES

Without TLS (plain HTTP):
   You ──"password=hunter2"──► [WiFi sniffer reads it!] ──► Server   exposed

With TLS (HTTPS):
   You ──"x9#@kL!8fP2q..."───► [sniffer sees gibberish] ──► Server   safe

WHERE TLS SITS

Application  (HTTP, WebSocket, SMTP...)
      |
   TLS   <- encrypts / decrypts + verifies identity here
      |
   TCP   <- reliable, ordered delivery
      |
   IP    <- routing across the network
Why "Transport Layer": it secures the transport, so any protocol can run on top — https://, wss:// (secure WebSocket), secure email, etc.

02The 3 Guarantees

GuaranteeMeaningStops
EncryptionData is scrambled; only the two ends can read itEavesdropping
AuthenticationYou're really talking to bank.com, not an impostorImpersonation / phishing
IntegrityData wasn't altered in transitTampering / injection
Forward secrecy (TLS 1.3): session keys are ephemeral (new per connection, then discarded). Even if the server's private key leaks later, an attacker can't decrypt old recorded traffic — that session's key is gone forever.

03Symmetric vs Asymmetric Encryption

TLS cleverly combines both. Understanding why is the key to the whole protocol.

Symmetric (e.g. AES)

ONE shared key both encrypts & decrypts. Like a physical key: same key locks and unlocks the door.

Fast — simple bit operations, often hardware-accelerated (AES-NI). Encrypts gigabytes/sec.

Weakness: both sides need the same key — how do you share it over an open network?

Asymmetric (e.g. RSA, ECC)

TWO keys — a public/private pair. Public encrypts, private decrypts (or signs). Like a mailbox: anyone drops mail in the slot, only you open it.

Slow — heavy math on 2048-bit+ numbers; ~100–1000× slower than AES.

Strength: can establish a secret safely over an open network.

THE GENIUS OF TLS — USE EACH FOR WHAT IT'S GOOD AT

Asymmetric solves "how do we agree on a key safely?"   (hard part, done ONCE)
Symmetric  solves "how do we encrypt tons of data fast?" (bulk work, ongoing)

Analogy:  send the key by ARMORED TRUCK once (asymmetric),
          then drive the SPORTS CAR for all traffic (symmetric).
Why not asymmetric always? Encrypting every byte with RSA would cripple performance (100–1000× slower). It's used only briefly, on tiny data, to set up the shared key.
Why not symmetric always? You can't share the secret key over an insecure network — an eavesdropper would just read it in transit. That's the key-distribution problem asymmetric crypto solves.

04The HTTPS Connection — Steps 1, 2, 3

A full HTTPS connection has three conceptual phases, all riding on an already-open TCP connection:

TCP 3-way handshake  (SYN / SYN-ACK / ACK)   <- opens the raw pipe first
      |
      v
+-----------------------------------------------------------+
| (1) KEY AGREEMENT     asymmetric + Diffie-Hellman          |
|     -> both sides derive the SAME secret session key      |
+-----------------------------------------------------------+
| (2) AUTHENTICATION    server presents its CERTIFICATE      |
|     -> browser verifies identity via the CA chain         |
+-----------------------------------------------------------+
| (3) DATA TRANSFER     fast symmetric AES with session key |
|     -> all HTTP traffic encrypted, cheaply                |
+-----------------------------------------------------------+
Note: in TLS 1.3 steps 1 & 2 happen together in a single round trip — the ClientHello/ServerHello exchange both agrees the key and carries the certificate. We split them here to explain each idea.

THE HANDSHAKE ON THE WIRE

Client                                          Server
  | -- ClientHello: TLS1.3, ciphers, my key-share --> |
  |                                                   |
  | <-- ServerHello: chosen cipher, my key-share ---- |
  |        + CERTIFICATE (signed by a CA)             |
  |                                                   |
  | (verify cert: CA-signed? right host? not expired?)|
  | (combine my private + your public -> SAME secret) |
  |                                                   |
  | ====== encrypted channel (AES-GCM) ============== |
  | -- GET /  (encrypted with session key) ---------> |
  | <-- 200 OK (encrypted) ------------------------- |

05Step 1 — Agreeing on a Key Without Sending It

The most beautiful trick in cryptography: both sides end up with the same secret key without ever transmitting it. An eavesdropper watching every byte still can't compute it. This is the Diffie-Hellman key exchange.

The paradox it solves — How can two people agree on a secret while shouting across a room full of eavesdroppers, such that the eavesdroppers don't learn the secret?

INTUITION — MIXING PAINT (easy to mix, impossible to un-mix)

1. Agree PUBLICLY on a common color:   YELLOW   (attackers see this)

2. Each picks a SECRET color (never shared):
   Client secret:  RED          Server secret:  BLUE

3. Mix secret with yellow, and SEND the mix:
   Client sends:  yellow+red  = ORANGE  -->
   Server sends:  yellow+blue = GREEN   <--      (attackers see these)

4. Add your OWN secret to what you RECEIVED:
   Client:  GREEN + own RED  = yellow+blue+red = BROWN
   Server:  ORANGE + own BLUE = yellow+red+blue = BROWN
            -> BOTH get the SAME BROWN = shared secret

Attacker saw yellow, orange, green — but to make brown they'd need to
UN-MIX a color back into red or blue. Separating mixed paint is infeasible.

THE ACTUAL MATH (modular exponentiation)

PUBLIC (everyone sees):  a big prime p  and a base g

Client picks secret a              Server picks secret b
Client computes A = g^a mod p      Server computes B = g^b mod p
        ----- sends A ----->              <----- sends B -----

Client computes  B^a mod p         Server computes  A^b mod p
   = (g^b)^a = g^(ab) mod p           = (g^a)^b = g^(ab) mod p
            |                                   |
            +------- SAME VALUE g^(ab) ---------+  = session key

Attacker knows p, g, A, B — but to get a from A = g^a mod p they must
solve the DISCRETE LOG problem: infeasible for 2048-bit+ numbers
(longer than the age of the universe on classical computers).
TLS uses ECDHE: Elliptic-Curve Diffie-Hellman Ephemeral — same idea with elliptic-curve math (smaller, faster, stronger), and a brand-new key per session (the "Ephemeral" part) → forward secrecy. The secrets a and b never leave their machines; only the "mixed" public values A and B travel.

06Step 2 — Certificates & the Chain of Trust

Diffie-Hellman agrees on a key but doesn't prove who you're talking to. Without identity, an attacker could do DH separately with each side (man-in-the-middle). The certificate fixes that.

Certificate = { domain, public key, expiry } + a digital signature from a trusted Certificate Authority (CA) that vouches for it. Anyone can verify the signature with the CA's public key. Only the CA (with its private key) can produce a valid one — so it can't be forged.

SIGNING vs VERIFYING (asymmetric, in reverse)

CA SIGNS:    hash(cert) encrypted with CA's PRIVATE key  = signature
YOU VERIFY:  decrypt signature with CA's PUBLIC key, compare to hash(cert)
   match    -> genuine + untampered + CA vouched for it   OK
   no match -> forged or altered                          REJECT

THE CHAIN OF TRUST

+---------------------------------------------------+
|  ROOT CA   (e.g. DigiCert Global Root)            |  self-signed;
|  private key kept OFFLINE in a vault (HSM)        |  PRE-INSTALLED on device
+------------------------+--------------------------+
                         | signs
                         v
+---------------------------------------------------+
|  INTERMEDIATE CA  (e.g. DigiCert TLS RSA CA)      |  signs day-to-day certs
+------------------------+--------------------------+
                         | signs
                         v
+---------------------------------------------------+
|  LEAF CERT  (bank.com)                            |  what the server presents
+---------------------------------------------------+
The anchor: your OS/browser ships with ~100–150 pre-installed root certificates (DigiCert, Let's Encrypt/ISRG, GlobalSign...). You trust bank.com because its cert chains up to a root you were "born trusting." The root is self-signed — trusted not by a signature but by being on that pre-vetted list.

HOW A SERVER GETS A CERT

  1. Server generates a key pair (keeps the private key secret).
  2. Sends a CSR (Certificate Signing Request) with its public key + domain to a CA.
  3. CA verifies domain control — Domain Validation (prove you own bank.com via a DNS record/file), or OV/EV (also verify the legal company).
  4. CA signs the cert with its private key → you install it on your server. (Let's Encrypt automates this via ACME — that's why HTTPS is now free.)

07How the Browser Verifies the Chain

Key fact: intermediate certificates are NOT shipped with your device. The server sends the full chain (leaf + intermediates) during the handshake. Only the root is pre-installed.

EACH CERT CONTAINS ITS OWN SUBJECT'S PUBLIC KEY

Leaf cert         contains -> bank.com's public key
Intermediate cert contains -> the Intermediate CA's public key   (server-sent)
Root cert         contains -> the Root CA's public key           (device-installed)

To verify cert X you need the ISSUER's public key — which lives in the
issuer's certificate, the next one up the chain.

VERIFICATION WALKS UP THE CHAIN

leaf         --verified by--> INTERMEDIATE's public key  (from server-sent cert)
intermediate --verified by--> ROOT's public key          (from device trust store)
root         --is it in my trusted store?--> YES  ->  whole chain trusted

You always need the key one level UP to verify the level below. The server
hands you every intermediate so the browser can climb until it lands on a
root that came pre-installed.

FULL VALIDATION CHECKLIST

  • Signature valid at every link (leaf → intermediate → root)
  • Chain ends at a root in my trusted store
  • Domain matches (cert says bank.com, I typed bank.com)
  • Not expired (within validity dates)
  • Not revoked (via CRL / OCSP revocation checks)
  • Correct purpose (server authentication)
Real-world bug — "missing intermediate": if a server is misconfigured and sends only the leaf, some browsers fail with "unable to verify the first certificate." That's why you always configure the full chain (fullchain.pem in Let's Encrypt), not just the leaf.
Why an attacker can't fake it: they can't sign a cert themselves (not a trusted CA), can't get a CA to sign for a domain they don't own, and can't steal a root's offline private key. No valid chain to a trusted root → connection refused.

08Step 3 — Fast Symmetric Data Transfer

Once the handshake establishes the shared session key and verifies identity, both sides switch to fast symmetric encryption (AES) for all the real traffic.

Handshake (asymmetric + DH)  -> establishes a shared SESSION KEY   (slow, once)
Data transfer (symmetric AES) -> encrypts ALL traffic              (fast, ongoing)

Client ── AES(session key, "GET /account") ──►  Server  (decrypts, reads plain HTTP)
Client ◄─ AES(session key, "200 OK ...") ─────  Server
Why this is the right design: AES is hardware-accelerated and encrypts gigabytes/sec, so HTTPS adds negligible overhead to the actual data. The expensive asymmetric math happened only once, on a few hundred bytes, to bootstrap this fast channel.

TIES BACK TO API GATEWAY & WEBSOCKETS

  • An API gateway terminates TLS — it decrypts public traffic, reads the plain HTTP, then routes (and often re-encrypts internally via mTLS).
  • Secure WebSockets (wss://) run over exactly this same TLS channel before the HTTP Upgrade → 101 handshake.

09Recap & Interview One-Liners

TLSThe protocol that encrypts + authenticates network traffic (the "S" in HTTPS). Uses a certificate + asymmetric handshake to verify the server and agree a shared secret, then fast symmetric AES for the data — giving encryption, authentication, and integrity over an untrusted network.
Symmetric vs asymmetricSymmetric (AES) is fast but can't safely share its key. Asymmetric is the reverse — safe key setup but far too slow for bulk data. TLS uses asymmetric once in the handshake to agree a key, then symmetric AES for all traffic.
Diffie-HellmanBoth sides pick a private secret, send only a "mixed" public value (g^a mod p), and combine the received value with their own secret to independently compute the same key (g^(ab) mod p) — which an eavesdropper can't reproduce because reversing modular exponentiation (discrete log) is infeasible. The key itself is never transmitted.
Certificate chainThe server's cert is signed by a CA, whose cert is signed up to a self-signed root pre-installed in your device. The server sends leaf + intermediates; the browser verifies each signature up to a trusted root. Only the root is the pre-shipped anchor.
